## 🔄 质量验证闭环流程（测试 → 开发 → 验证）

```
开发完成  →  自测通过  →  提测
   ↓
测试工程师执行测试 → 记录 Bug → 分配给对应开发
   ↓
开发修复 Bug → 标记为"待验证" → 通知测试
   ↓
测试验证 → 通过 → 关闭 Bug
   ↓ 不通过 → 重新打开 → 返回开发修复
   ↓
循环直到所有 P0/P1 Bug 关闭
   ↓
财务对账验证 → 绩效核算验证 → UI 验证 → 最终验收
```

### 缺陷分级标准

| 级别 | 定义 | 处理要求 |
|------|------|----------|
| P0 | 阻断性缺陷，核心流程无法进行 | 必须修复才能进入下一阶段 |
| P1 | 重要功能缺陷，影响用户使用 | 必须在当前 Sprint 修复 |
| P2 | 一般功能缺陷，不影响核心流程 | 可安排在下一 Sprint 修复 |
| P3 | 美化优化建议，不影响使用 | 可安排在下一 Sprint 处理 |

### 验证流程责任

| 环节 | 责任人 | 输出 |
|------|--------|------|
| 模块开发完成 | 开发工程师 | 自测通过，单元测试覆盖 |
| 集成测试 | 测试工程师 | 测试报告，Bug 列表 |
| Bug 修复 | 对应开发 | Bug 修复完成，待验证 |
| 回归验证 | 测试工程师 | Bug 验证通过/不通过 |
| 财务对账 | 财务专员 | 对账报告，数据准确确认 |
