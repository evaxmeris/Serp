## 1. 当前问题分析

### 1.1 现状

目前 Trade ERP 开发过程中，**所有工作都由主代理完成**，存在以下问题：

1. **上下文过载**：主代理需要同时处理任务分析、代码编写、测试验证、文档编写等所有工作，上下文窗口容易耗尽
2. **效率低下**：串行处理多个任务，无法并行开发
3. **焦点分散**：主代理需要在不同类型的任务之间频繁切换，影响专注度
4. **可扩展性差**：无法支持大规模并行开发，特别是 Sprint 测试阶段需要多任务同时进行

### 1.2 典型场景痛点

**痛点案例 1：Sprint 测试阶段**
- 需要同时进行：前端修复、后端修复、测试验证、文档更新
- 串行处理导致测试周期延长
- 主代理上下文不断切换，容易遗漏细节

**痛点案例 2：BUG 修复阶段**
- 多个独立 BUG 需要修复
- 每个 BUG 都需要重新加载上下文
- 修复完成后需要逐个验证，效率低下

## 2. 子代理使用规范

### 2.1 什么任务应该分配给子代理

#### ✅ 推荐分配给子代理的任务

| 任务类型 | 说明 | 示例 |
|---------|------|------|
| **独立代码修复** | 单个独立的 BUG 修复，不涉及大范围重构 | 修复移动端按钮点击无响应、修复某个 API 返回错误 |
| **功能模块开发** | 相对独立的功能模块，可以并行开发 | 开发报价单导出功能、开发客户列表分页 |
| **测试任务** | 独立的测试验证工作 | 测试登录功能、测试报价单创建流程 |
| **文档编写** | 项目文档、测试报告、总结文档编写 | 编写 Sprint 测试报告、编写 API 文档 |
| **代码审查** | 对特定模块进行代码审查 | 审查 P0 修复代码质量、检查安全问题 |
| **性能优化** | 独立的性能优化任务 | 优化图片加载速度、优化数据库查询 |

#### ⚠️ 谨慎分配的任务

| 任务类型 | 说明 | 建议 |
|---------|------|------|
| **架构重构** | 涉及多个模块的大规模重构 | 主代理主导，可以分解为多个子任务分配给子代理 |
| **核心逻辑修改** | 影响系统稳定性的核心逻辑变更 | 主代理审查设计后，子代理执行 |
| **跨模块依赖任务** | 任务之间有强依赖关系 | 主代理协调依赖顺序，子代理执行 |
| **需要交互式调试** | 需要频繁与用户交互确认的任务 | 主代理处理 |

#### ❌ 不应该分配给子代理的任务

- 需求分析和架构决策（需要整体上下文）
- 直接与用户交互沟通（主代理负责）
- 环境配置和基础设施搭建（主代理负责）
- 敏感操作（部署到生产、数据库迁移等，主代理负责）

### 2.2 如何组织和协调多个子代理

#### 2.2.1 协调原则

1. **任务隔离**：每个子代理只负责一个独立任务，职责清晰
2. **无依赖并行**：相互独立的任务同时分派，并行执行
3. **依赖串行**：有依赖关系的任务按顺序执行，前一个完成后再分派下一个
4. **结果汇总**：所有子代理完成后，主代理汇总结果，进行整体验证

#### 2.2.2 组织模式

**模式 A：并行无依赖（推荐）**

```
主代理
├── 子代理 1：修复前端 BUG-001
├── 子代理 2：修复后端 BUG-002
├── 子代理 3：执行测试用例 A
└── 子代理 4：编写测试报告
     ↓
所有完成 → 主代理汇总验证
```

**适用场景**：多个独立 BUG 修复、多个独立测试任务、文档编写等

**模式 B：依赖流水线**

```
主代理
└─→ 子代理 1：修复后端 API
    ↓
    完成 → 子代理 2：修复前端调用
    ↓
    完成 → 子代理 3：集成测试
    ↓
    完成 → 主代理验证
```

**适用场景**：前后端联动的修复、需要按步骤执行的任务

**模式 C：分层协同**

```
主代理：任务分解 → 分配子任务
├── 前端子代理组（3 个子代理并行修复前端）
├── 后端子代理组（2 个子代理并行修复后端）
└── 测试子代理（等待修复完成后执行测试）
```

**适用场景**：Sprint 大规模测试修复阶段

### 2.2.3 协调流程

```
1. 主代理接收任务
   ↓
2. 任务分解
   - 将大任务分解为多个独立子任务
   - 识别任务依赖关系
   ↓
3. 创建任务描述
   - 使用标准模板
   - 明确目标、范围、验收标准
   ↓
4. 分派子代理
   - 无依赖任务：并行分派所有
   - 依赖任务：按顺序分派
   ↓
5. 等待完成
   - 不轮询，等待自动通知
   ↓
6. 结果汇总
   - 收集所有子代理输出
   - 检查每个任务完成状态
   ↓
7. 整体验证
   - 运行整体测试
   - 确认所有任务达标
   ↓
8. 提交/部署
```

### 2.2.4 冲突解决

- **代码冲突**：主代理负责合并解决，子代理只修改自己负责的文件
- **需求冲突**：主代理负责澄清和决策
- **质量不达标**：退回子代理重新修改，或主代理自行修正

## 3. 任务分配模板

### 3.1 标准模板

```markdown
# 任务名称

## 任务目标
[清晰描述要完成什么]

## 任务背景
[为什么需要做这个任务，相关上下文]

## 工作范围
### 需要修改的文件
- 文件路径 1
- 文件路径 2
- ...

### 不修改的文件（明确边界）
- 文件路径 A
- ...

## 验收标准
[可验证的完成标准，每条用 ✓ 表示]
- ✓ [标准 1]
- ✓ [标准 2]

## 约束条件
- 必须保持向后兼容
- 禁止删除现有字段/API
- ...

## 参考文档
- 相关文档链接
- 相关 BUG 描述
```

### 3.2 模板说明

1. **任务目标**：一句话说清楚要做什么
2. **任务背景**：提供必要的上下文，帮助子代理理解
3. **工作范围**：**必须**明确边界，哪些改哪些不改，防止子代理越界
4. **验收标准**：必须可验证，完成就是完成，没完成就是没完成
5. **约束条件**：特别重要的限制，比如必须向后兼容，不能删除字段等
6. **参考文档**：相关文档链接，帮助子代理快速理解

## 5. 工作流程总结

### 5.1 主代理职责

1. **任务接收与分解**：接收用户需求，分解为可分配的子任务
2. **任务描述编写**：使用标准模板编写清晰的任务描述
3. **子代理分派**：根据依赖关系选择并行或串行分派
4. **结果汇总验证**：收集所有结果，进行整体验证
5. **质量把关**：确保输出质量符合要求
6. **用户沟通**：向用户汇报结果

### 5.2 子代理职责

1. **按任务描述执行**：只完成分配的任务，不越界
2. **遵守约束条件**：特别是向后兼容等重要约束
3. **输出清晰结果**：完成后说明完成了什么，结果如何
4. **发现问题及时报告**：遇到问题或疑问，立即报告主代理
5. **不擅自决策**：超出任务范围的问题请示主代理

### 5.3 最佳实践

1. **小任务优先**：任务拆得越小，子代理执行效率越高
2. **边界清晰**：明确哪些改哪些不改，减少误解
3. **上下文充分**：提供必要的参考文档链接
4. **验收标准可验证**：避免模糊描述，完成就是完成
5. **并行执行**：无依赖任务尽可能并行，节省时间
