## 一、最初技术路线选择 (NextAuth)

### 1.1 选择 NextAuth 的原因

| 考量维度 | 决策依据 |
|----------|----------|
| **生态成熟度** | NextAuth 是 Next.js 官方推荐认证方案，社区活跃 |
| **功能完整性** | 支持多种 Provider (Credentials, OAuth, SSO) |
| **安全性** | 内置 CSRF 保护、JWT 签名、Session 管理 |
| **开发效率** | 配置式开发，减少样板代码 |
| **未来扩展** | 支持后续添加 GitHub/Google 第三方登录 |

### 1.2 技术栈假设

```
Next.js 15.x + NextAuth v4.x = 稳定组合
```

**问题**: 项目实际使用 **Next.js 16.1.6**，打破了原有假设

---

## 二、遇到的问题

### 2.1 NextAuth v4.x 问题

| 问题 | 表现 | 根因 |
|------|------|------|
| CSRF 验证失败 | 登录后重定向到 `/api/auth/signin?csrf=true` | Next.js 16 异步 Cookies API 与 NextAuth v4 不兼容 |
| `cookies().getAll` 错误 | `TypeError: cookies(...).getAll is not a function` | Next.js 16 使用异步 API，NextAuth v4 期望同步 API |
| `params.nextauth` 错误 | `params is a Promise` | Next.js 16 动态路由 params 改为 Promise |

### 2.2 NextAuth v5 Beta 问题

| 问题 | 表现 | 根因 |
|------|------|------|
| authorize 未调用 | 日志显示"缺少 email 或 password" | Credentials 数据传递机制变更，表单数据未正确解析 |
| Beta 稳定性 | API 可能变更，文档不完善 | v5 仍在 Beta 阶段 |

## 三、简化认证方案分析

### 3.1 技术实现

```typescript
// 核心逻辑
1. 登录 API: POST /api/auth/login
   - 验证用户 credentials
   - 生成 JWT token
   - 设置 HttpOnly Cookie

2. 获取用户 API: GET /api/auth/me
   - 验证 JWT token
   - 返回用户信息

3. 登出 API: POST /api/auth/logout
   - 删除 Cookie
```

### 3.2 优势

| 维度 | 简化方案 | NextAuth |
|------|----------|----------|
| **兼容性** | ✅ 完全兼容 Next.js 16 | ❌ 需要等待 v5 稳定 |
| **可控性** | ✅ 代码完全可控，易调试 | 🟡 黑盒，调试困难 |
| **依赖** | ✅ 仅依赖 `jose` (轻量) | 🟡 依赖 next-auth (重量级) |
| **学习曲线** | ✅ 简单直观 | 🟡 需要学习 NextAuth 概念 |
| **上线时间** | ✅ 1-2 小时 | ❌ 不确定 (可能数天) |
| **安全性** | ✅ JWT + HttpOnly Cookie | ✅ 内置安全机制 |
| **功能完整性** | 🟡 基础认证 | ✅ 完整认证生态 |

### 3.3 劣势

| 维度 | 简化方案 | NextAuth |
|------|----------|----------|
| **第三方登录** | ❌ 需要手动实现 | ✅ 开箱即用 |
| **多 Provider** | ❌ 需要手动扩展 | ✅ 配置即可 |
| **社区支持** | 🟡 自定义方案 | ✅ 大量文档/示例 |
| **安全审计** | 🟡 自行负责 | ✅ 经过社区验证 |
| **功能扩展** | 🟡 自行开发 | ✅ 插件生态 |

## 四、推荐方案

### 4.1 短期方案 (当前迭代)

**采用简化认证方案**

**理由**:
1. **时间紧迫** - 系统需要尽快可用
2. **风险可控** - 核心功能简单，易测试
3. **可逆** - 后续可替换为 NextAuth

**实施计划**:
```
Day 1: 实现基础登录/登出/会话管理
Day 2: 集成到现有页面，测试验证
Day 3: 安全加固 (速率限制、密码策略)
```

### 4.2 长期方案 (v1.0 之前)

**评估后决定**:

| 条件 | 推荐方案 |
|------|----------|
| 需要第三方登录 | 升级 NextAuth v5 (稳定版) |
| 仅需账号密码 | 保持简化方案 |
| 需要 SSO/SAML | 引入专业认证服务 (Auth0/Clerk) |

## 五、安全考量

### 5.1 简化方案安全措施

| 风险 | 缓解措施 |
|------|----------|
| JWT 泄露 | HttpOnly Cookie + Secure 标志 |
| 暴力破解 | 登录失败次数限制 |
| 会话劫持 | JWT 过期时间 (7 天) + 刷新机制 |
| XSS 攻击 | Content-Security-Policy + 输入验证 |
| CSRF 攻击 | SameSite=strict + 自定义 header |

### 5.2 待实现安全功能

- [ ] 登录失败速率限制 (每 5 次失败锁定 15 分钟)
- [ ] 密码强度策略 (长度/复杂度)
- [ ] 双因素认证 (可选)
- [ ] 会话管理 (查看/撤销活跃会话)
- [ ] 审计日志 (登录/登出/权限变更)

## 六、决策建议

### 6.1 立即决策

**采用简化认证方案**，确保系统可用

### 6.2 技术债务管理

| 项目 | 优先级 | 时间窗 |
|------|--------|--------|
| 速率限制 | P1 | 1 周内 |
| 密码策略 | P1 | 1 周内 |
| 审计日志 | P2 | 2 周内 |
| 双因素认证 | P3 | v1.0 前 |

### 6.3 退出策略

如简化方案不满足需求，可迁移至:
- **Auth0** - 企业级认证服务
- **Clerk** - 开发者友好的认证平台
- **NextAuth v5** - 待稳定版本发布
