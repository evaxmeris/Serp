## 1. 设计原则

### 1.1 RESTful 规范

- 使用资源名词，不使用动词
- 使用 HTTP 动词语义正确
  - `GET`: 获取资源
  - `POST`: 创建资源
  - `PUT`: 更新资源
  - `DELETE`: 删除资源
- 路径使用小写，单词间用连字符 (`kebab-case`)
- 版本号包含在路径中: `/api/v1/`

### 1.2 查询参数命名

- 使用 `camelCase`
- 分页参数统一: `page`, `limit`
- 搜索参数统一: `search`
- 排序参数统一: `sortBy`, `sortOrder`

### 1.3 请求体命名

- 使用 `camelCase`
- 必填参数前端验证 + 后端验证双重验证

## 2. 统一响应格式

### 2.1 成功响应

```typescript
{
  "success": true,
  "code": "SUCCESS",
  "data": T,
  "message": "操作成功",
  "timestamp": "2026-03-23T10:00:00Z"
}
```

### 2.2 列表响应

```typescript
{
  "success": true,
  "code": "SUCCESS",
  "data": {
    "items": T[],
    "pagination": {
      "page": 1,
      "limit": 20,
      "total": 100,
      "totalPages": 5
    }
  },
  "message": "查询成功",
  "timestamp": "2026-03-23T10:00:00Z"
}
```

### 2.3 创建成功响应

```typescript
{
  "success": true,
  "code": "CREATED",
  "data": T,
  "message": "创建成功",
  "timestamp": "2026-03-23T10:00:00Z"
}
```
- HTTP 状态码: `201 Created`

### 2.4 删除成功响应

- HTTP 状态码: `204 No Content`
- 无响应体

### 2.5 错误响应

```typescript
{
  "success": false,
  "code": "ERROR_CODE",
  "message": "错误描述",
  "errors": [
    { "field": "email", "message": "邮箱格式不正确" }
  ],
  "timestamp": "2026-03-23T10:00:00Z"
}
```

### 2.6 HTTP 状态码对照表

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| 200 | OK | 成功获取或更新 |
| 201 | Created | 资源创建成功 |
| 204 | No Content | 删除成功 |
| 400 | Bad Request | 请求参数错误 |
| 401 | Unauthorized | 未认证 |
| 403 | Forbidden | 无权限 |
| 404 | Not Found | 资源不存在 |
| 409 | Conflict | 业务冲突 |
| 422 | Unprocessable Entity | 参数验证失败 |
| 500 | Internal Server Error | 服务器错误 |

## 3. 错误码规范

### 3.1 通用错误码

| 错误码 | HTTP 状态 | 说明 |
|--------|----------|------|
| `SUCCESS` | 200 | 成功 |
| `CREATED` | 201 | 创建成功 |
| `NO_CONTENT` | 204 | 删除成功 |
| `BAD_REQUEST` | 400 | 请求参数错误 |
| `UNAUTHORIZED` | 401 | 未认证 |
| `FORBIDDEN` | 403 | 无权限 |
| `NOT_FOUND` | 404 | 资源不存在 |
| `CONFLICT` | 409 | 资源冲突 |
| `VALIDATION_ERROR` | 422 | 验证失败 |
| `INTERNAL_ERROR` | 500 | 服务器错误 |

## 4. 版本管理

### 4.1 版本策略

- 路径版本化: `/api/v1/`, `/api/v2/`
- 向后兼容至少 2 个大版本
- 废弃接口提前 3 个月通知

### 4.2 当前版本

| 版本 | 模块 | 状态 |
|------|------|------|
| 无版本 | auth/customers/inquiries/quotations/orders/products/suppliers/purchases | 保留，逐步迁移到 v1 |
| v1 | inbound-orders/outbound-orders/inventory/reports/purchase-orders | 当前稳定版本 |

### 4.3 废弃流程

1. 在代码中标记接口为 `@deprecated`
2. 在响应头添加 `Deprecation` 字段
3. 文档中标注废弃日期和替代方案
4. 到期后移除接口

#### 5.2.4 确认入库

```http
POST /api/v1/inbound-orders/:id/confirm
```

**请求体:**
```json
{
  "actualDate": "2026-03-23",
  "items": [
    {
      "inboundOrderItemId": "item-xxx",
      "actualQuantity": 980,
      "batchNo": "BATCH202603",
      "productionDate": "2026-03-01",
      "expiryDate": "2027-03-01"
    }
  ],
  "note": "实际入库 980 件，20 件短缺"
}
```

**业务流程:**
1. 验证入库单状态为 `PENDING`
2. 更新入库单实际数量
3. 逐项更新 `InventoryItem` 库存
4. 生成 `StockMovement` 记录
5. 生成 `InventoryLog` 记录
6. 如果关联采购单，更新收货进度
7. 更新状态为 `COMPLETED` (全部入库) 或 `PARTIAL` (部分入库)

#### 5.2.5 取消入库

```http
POST /api/v1/inbound-orders/:id/cancel
```

**请求体:**
```json
{
  "cancelReason": "采购单取消"
}
```

**业务规则:** 只能取消 `PENDING` 状态的入库单

| warehouseId | string | 否 | 仓库筛选 |

```http
POST /api/v1/outbound-orders
```

**请求体:**
```json
{
  "orderId": "so-xxx",
  "warehouseId": "wh-xxx",
  "shipmentId": "ship-xxx",
  "note": "订单发货出库",
  "items": [

    }
  ]

```

**业务规则:**
- 创建前检查库存是否充足
- 库存不足返回 `INSUFFICIENT_STOCK` 错误
- 创建后状态为 `PENDING`

#### 5.3.3 批量创建出库单

```http
POST /api/v1/outbound-orders/batch
```

**说明:** 合并发货时，一次性为多个订单创建出库单

**请求体:**
```json
{
  "orders": [
    {
      "orderId": "so-xxx",
      "warehouseId": "wh-xxx",
      "items": [...]
    }
  ]
}
```

#### 5.3.4 确认出库

```http
POST /api/v1/outbound-orders/:id/confirm
```

**业务流程:**
1. 验证状态为 `PENDING`
2. 再次检查库存充足
3. 逐项扣减 `InventoryItem` 库存
4. 生成 `StockMovement`
5. 生成 `InventoryLog`
6. 更新订单项 `shippedQty`
7. 状态变为 `COMPLETED`

#### 5.3.5 取消出库

```http
POST /api/v1/outbound-orders/:id/cancel
```

**业务规则:**
- 只能取消 `PENDING` 状态
- 已扣减的库存会返还

### 6.2 利润分析报表

```http
GET /api/v1/reports/profit
```

**查询参数:**
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| startDate | string | 是 | 开始日期 |
| endDate | string | 是 | 结束日期 |
| groupBy | string | 否 | 分组维度: order/product/category/month |

**计算逻辑:**
- 销售收入 = sum(订单明细金额)
- 销售成本 = sum(数量 × 产品成本价)
- 毛利 = 销售收入 - 销售成本
- 毛利率 = 毛利 / 销售收入 × 100%

Authorization: Bearer <jwt_token>

### 7.3 数据权限

- **Admin**: 看到所有数据
- **Manager**: 看到所有数据
- **User**: 只能看到自己负责的数据（自己创建的订单/采购等）
- **Viewer**: 只能查看，不能修改

## 8. 速率限制

响应头包含速率限制信息:

```http
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1709712000
```

**限制标准:**

| API 类型 | 限制 |
|----------|------|
| 普通查询 | 100 次/分钟 |
| 报表导出 | 5 次/分钟 |
| 认证接口 | 10 次/分钟 |
| 批量操作 | 20 次/分钟 |
