## 3. 认证授权架构

### 3.1 整体架构: JWT + RBAC

```
┌─────────────────────────────────────────────────────┐
│                         客户端                        │
└────────────┬────────────────────────────────────────┘
             │  Authorization: Bearer <JWT>
             ▼
┌─────────────────────────────────────────────────────┐
│                   Next.js API Route                  │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│                 JWT Authentication Middleware        │
│  - 验证 JWT 签名                                     │
│  - 解析用户 ID → User 对象                           │
│  - 注入到 Request context                             │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│                 RBAC Authorization Middleware        │
│  - 从 User 加载所有角色                               │
│  - 从所有角色加载所有权限                             │
│  - 检查 当前路由 是否在权限范围内                       │
└────────────┬────────────────────────────────────────┘
             │  通过
             ▼
┌─────────────────────────────────────────────────────┐
│                   业务处理 Controller                │
└─────────────────────────────────────────────────────┘
```

### 3.2 RBAC 模型设计

**模型**: User ←(M:N)→ Role ←(M:N)→ Permission

- **User**: 系统用户
- **Role**: 角色，代表一组权限的集合
- **Permission**: 细粒度权限，格式: `模块:操作:资源`

### 3.3 权限编码规范

```
格式: <module>:<action>:<resource>?

示例:
- system:user:list          - 查看用户列表
- system:user:create        - 创建用户
- system:user:update        - 更新用户
- system:user:delete        - 删除用户
- order:platform:sync       - 同步平台订单
- warehouse:overseas:manage - 管理海外仓
- finance:transaction:view  - 查看财务交易
- finance:reimbursement:approve - 审批报销
```

### 3.4 默认角色权限配置

| 角色 | 角色编码 | 说明 | 默认权限 |
|------|---------|------|---------|
| 系统管理员 | `admin` | 系统超级管理员 | 全部权限 |
| 运营经理 | `manager` | 运营经理 | 除系统管理外全部权限 |
| 运营专员 | `user` | 普通运营 | 查看+创建订单产品等 |
| 财务人员 | `finance` | 财务人员 | 财务模块权限 |
| 访客 | `viewer` | 只读访客 | 只读权限 |

### 3.5 JWT 配置

- **签名算法**: RS256（非对称加密，更安全）
- **Access Token 有效期**: 24 小时
- **Refresh Token**: 支持刷新
- **Token 存储**: HttpOnly Cookie（防 XSS）

### 3.6 中间件实现

```typescript
// 伪代码
async function authMiddleware(req: NextRequest) {
  const token = extractToken(req);
  if (!token) return 401;

  const payload = verifyJWT(token);
  const user = await prisma.user.findUnique({ where: { id: payload.userId } });
  if (!user || user.status !== ACTIVE) return 401;

  // 加载用户所有权限
  const permissions = await getUserPermissions(user.id);
  req.context = { user, permissions };
}

function requirePermission(permission: string) {
  return (req) => {
    if (!req.context.permissions.includes(permission)) {
      return 403;
    }
  };
}
```

## 5. 海外仓同步机制

### 5.1 支持的海外仓类型

| 类型 | 说明 | 同步方式 |
|------|------|---------|
| FBA | Amazon FBA 仓库 | API 拉取库存报告 |
| 第三方海外仓 | 第三方服务商提供的仓库 | API 对接 / CSV 导入 |
| 自运营海外仓 | 自管 | ERP 内部管理 |

### 5.2 数据模型

```
OverseasWarehouse        ← 1:N → OverseasInventory
OverseasInventory        ← 1:N → OverseasInventoryLog
OverseasWarehouse        ← 1:N → OverseasShipmentPlan
OverseasShipmentPlan     ← 1:N → OverseasShipmentPlanItem
```

### 5.3 FBA 库存同步流程

```
Amazon FBA 同步流程:

  1. 定期调用 Selling Partner API
  2. 获取 FBA 库存报告 (GET_FBA_INVENTORY_STATISTICS_DATA)
  3. 解析报告:
     - SKU → 关联本地 Product
     - totalQty, afnFulfillableQty → availableQty
     - afnInboundWorkingQty → inboundQty
  4. 更新或创建 OverseasInventory 记录
  5. 记录变动日志 (OverseasInventoryLog)

频率: 每天 2 次 (或手动触发)
```

### 5.4 FBA 发货计划流程

```
发往 FBA 流程:

  ┌─────────────────────────────────────────────────┐
  │  1. 创建发货计划草稿                               │
  │  2. 添加商品和数量                                 │
  │  3. 计算总重量 / 预估运费                          │
  │  4. 提交 → 在 Amazon 创建计划                      │
  │  5. 获取 Amazon 提供的箱唛 PDF                     │
  │  6. 发货后 → 更新状态为 IN_TRANSIT                 │
  │  7. FBA 签收后 → 更新库存 + 更新状态为 RECEIVED    │
  └─────────────────────────────────────────────────┘
```

### 5.5 库存计算逻辑

```
availableQty = totalQty - reservedQty - inboundQty?
```

所有库存变动必须记录日志，便于追溯对账。

### 5.6 国内仓与海外仓的关系

- 国内库存: `inventory_items` (原有表)
- 海外库存: `overseas_inventory` (新增表)
- 独立存储，独立计算，支持多维度库存查看
- 支持库存调拨: 国内 → 海外仓

## 6. 审批流程引擎设计

### 6.1 设计目标

- 支持自定义审批流程 (不同单据类型不同流程)
- 支持多级审批
- 支持审批历史追溯
- 轻量化，不做过度设计

### 6.2 模型设计

```
ApprovalFlow        ← 1:N → ApprovalRecord
ApprovalRecord      ← 1:N → ApprovalHistory

ApprovalFlow: 流程定义 (模板)
  - 配置多个节点 nodes[]
  - 每个节点: 类型(用户/角色/上级) + 审批人

ApprovalRecord: 审批实例
  - 当前步骤 currentStep
  - 当前审批人
  - 状态
  - 关联业务实体 (entityType + entityId)

ApprovalHistory: 审批历史记录
  - 每一步审批都保存记录
  - 审批人、动作、时间、意见
```

### 6.3 节点类型

| 类型 | 说明 |
|------|------|
| `USER` | 指定固定用户审批 |
| `ROLE` | 指定角色任意一人审批 |
| `SUPERVIOR` | 申请人直接上级 |
| `MANAGER` | 部门经理 |
| `CEO` | CEO |

### 6.4 审批流转逻辑

```typescript
// 伪代码: 提交审批
function startApproval(flow: ApprovalFlow, entity: any, creator: User) {
  // 1. 创建审批记录
  const record = createApprovalRecord(flow, entity, creator);

  // 2. 获取第一个节点
  const firstNode = flow.nodes[0];

  // 3. 确定当前审批人
  const approverId = resolveApprover(firstNode, creator);

  // 4. 更新记录，发送通知
  record.currentStep = 1;
  record.approverId = approverId;
  sendApprovalNotification(approverId, record);

  return record;
}

// 伪代码: 处理审批
function processApproval(
  record: ApprovalRecord,
  history: ApprovalHistory,
  action: ApprovalAction
) {
  // 1. 保存审批历史
  saveHistory(record, history);

  // 2. 如果拒绝/打回 → 流程结束
  if (action === REJECTED || action === RETURNED) {
    record.status = action;
    record.finishedAt = now();
    notifyCreator(record);
    return;
  }

  // 3. 如果通过且还有下一步 → 流转到下一节点
  if (action === APPROVED && record.currentStep < record.totalSteps) {
    const nextStep = record.currentStep + 1;
    const nextNode = flow.nodes[nextStep - 1];
    const nextApprover = resolveApprover(nextNode, record.createdBy);
    record.currentStep = nextStep;
    record.approverId = nextApprover;
    sendApprovalNotification(nextApprover, record);
    return;
  }

  // 4. 如果是最后一步 → 流程结束
  if (action === APPROVED && record.currentStep === record.totalSteps) {
    record.status = APPROVED;
    record.finishedAt = now();
    // 执行业务回调 (如更新订单状态为已批准)
    executeApprovalCallback(record);
    notifyCreator(record);
    return;
  }
}
```

## 7. 财务模块架构

### 7.1 模块范围

- 账户管理 (现金、银行、支付宝、微信、PayPal 等)
- 交易明细记录 (收入/支出/转账)
- 费用管理与报销流程
- 与业务单据关联 (订单/采购单/报销单)
- 对帐功能

### 7.2 对账模型

```
Account (账户)
  ↓ 1:N
Transaction (交易明细)
  - 每笔交易改变账户余额
  - 支持对帐标记 (reconciled)
  - 关联业务单据 (referenceType + referenceId)

Expense (费用)
  ↓ N:1
Reimbursement (报销单)
  - 一个报销单包含多个费用
  - 走审批流程
  - 审批通过后付款 → 生成 Transaction
```

### 7.3 余额更新流程

**事务保证**: 所有余额更新必须在事务中执行

```typescript
// 创建交易流程:
await prisma.$transaction(async (tx) => {
  // 1. 锁定账户行
  const account = await tx.account.findUnique({
    where: { id: accountId },
  });
  // 2. 计算新余额
  const newBalance = calculateNewBalance(account.currentBalance, transaction);
  // 3. 更新账户余额
  await tx.account.update({
    where: { id: accountId },
    data: { currentBalance: newBalance },
  });
  // 4. 创建交易记录
  await tx.transaction.create({
    data: {
      ...transaction,
      balanceAfter: newBalance,
    },
  });
});
```

### 7.4 多币种支持

- 每个账户有默认币种
- 交易记录保存原币金额和汇率
- 报表支持本位币折算

### 7.5 报表支持

基于现有的 `report_definitions` 和 `report_data` 框架:

预定义财务报表:

1. **现金流量表** - 按期间统计收支
2. **收支分类统计表** - 按分类统计
3. **账户余额表** - 各账户当前余额
4. **费用报销统计** - 按人员/部门统计

## 8. API 设计规范

### 8.1 RESTful 规范

**统一前缀**: `/api/v1`

**资源命名**: 复数，kebab-case

```
GET    /api/v1/platform-accounts      - 列表
GET    /api/v1/platform-accounts/:id  - 详情
POST   /api/v1/platform-accounts      - 创建
PUT    /api/v1/platform-accounts/:id  - 更新
DELETE /api/v1/platform-accounts/:id  - 删除

POST   /api/v1/platform-accounts/:id/sync-orders - 触发同步
```

### 8.2 响应格式

```json
{
  "success": true,
  "data": { ... },
  "message": "说明信息",
  "meta": {
    "page": 1,
    "pageSize": 20,
    "total": 100,
    "totalPages": 5
  }
}
```

错误响应:

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "错误描述"
  }
}
```

### 8.3 分页

查询参数:
- `page` - 页码，从 1 开始
- `pageSize` - 每页条数
- `sortBy` - 排序字段
- `sortOrder` - `asc`/`desc`

### 8.4 权限控制

所有 API 统一通过中间件检查权限:

```typescript
// 示例
GET /api/v1/finance/transactions
  → 需要权限: finance:transaction:list
```

## 9. 部署架构建议

### 9.1 开发环境

```
┌─────────────┐
│  Next.js    │  localhost:3000
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ PostgreSQL  │  localhost:5432
└─────────────┘
```

### 9.2 生产环境推荐 (Docker Compose)

```
┌─────────────────────────────────────────────────────┐
│                   Nginx 反向代理                      │
│ - SSL 终止                                           │
│ - 静态资源缓存                                        │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│              Next.js 应用容器 (多实例)                │
│              Node.js 运行时                           │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│                   PostgreSQL 数据库                   │
│ - 持久化存储                                          │
│ - 定期备份                                           │
└─────────────────────────────────────────────────────┘
```

### 9.3 定时任务

使用 Node.js node-cron 模块，在应用内运行:

```
*/15 * * * *  → 多平台订单同步
0 */12 * * *  → FBA 库存同步
0 1 * * *     → 生成日报表
```

注意: 如果多实例部署，需要使用分布式锁避免重复执行。

### 9.4 监控

- **健康检查**: `/api/health`
- **日志**: JSON 格式，可接入 ELK
- **错误追踪**: Sentry

## 10. 安全性考虑

### 10.1 认证授权

- ✅ JWT 非对称签名 (RS256)
- ✅ HttpOnly Cookie 存储 Token
- ✅ 密码 bcrypt 哈希存储
- ✅ 细粒度 RBAC 权限控制

### 10.2 敏感数据保护

- ✅ 平台 API 密钥/Token 加密存储
- ✅ 敏感字段不返回给前端
- ✅ 数据库连接字符串使用环境变量
- ✅ 不提交敏感信息到 Git

### 10.3 API 安全

- ✅ 输入验证 (Zod 或 class-validator)
- ✅ 防 SQL 注入 (Prisma ORM 已处理)
- ✅ CORS 配置允许的 Origin
- ✅ 限流防止暴力攻击

### 10.4 审批安全

- ✅ 检查审批人是否有权限审批
- ✅ 不可越权审批
- ✅ 完整审批日志追溯

## 11. 性能考虑

### 11.1 数据库优化

- ✅ 外键字段都添加索引
- ✅ 常用查询字段添加复合索引
- ✅ 分页使用游标查询/offset 分页
- ✅ 大数据量表 (交易日志) 按时间分区 (可选)

### 11.2 应用层优化

- **列表查询**: 限制最大返回条数
- **同步任务**: 异步处理，返回 Task ID 轮询状态
- **缓存**: 权限列表可缓存 (LRU Cache)
- **图片**: 使用 CDN 托管

### 11.3 并发控制

- 使用数据库事务保证数据一致性
- 余额更新悲观锁 (SELECT FOR UPDATE)
- 同步操作分布式锁防止并发

## 12. 扩展性考虑

### 12.1 平台扩展性

- 新增平台只需实现 PlatformService 接口
- 适配器模式，不影响现有代码
- 平台类型枚举可扩展

### 12.2 功能扩展性

- 审批流程可配置，无需改代码
- 权限可动态分配
- 角色可自定义

### 12.3 业务增长

- 数据库可水平分库分表 (按时间/业务)
- 应用无状态，可水平扩展
- 异步任务可迁移到独立队列服务 (BullMQ)
