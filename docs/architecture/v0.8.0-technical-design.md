## 2. 数据库设计

### 2.1 新增表

#### 2.1.1 `roles` - 角色表

```prisma
model Role {
  id          String          @id @default(cuid())
  name        String          // 角色名称，如 "销售经理"
  code        String          @unique  // 角色编码，如 "sales_manager"
  description String?         // 角色描述
  isSystem    Boolean         @default(false)  // 是否系统内置角色
  isActive    Boolean         @default(true)
  permissions Json            @default("[]")  // 权限列表
  createdAt   DateTime        @default(now())
  updatedAt   DateTime        @updatedAt
  users       UserRole[]
  permissions Permission[]

  @@index([code])
  @@index([isActive])
  @@map("roles")
}
```

#### 2.1.2 `user_roles` - 用户角色关联表

```prisma
model UserRole {
  id        String    @id @default(cuid())
  userId    String
  roleId    String
  createdAt DateTime  @default(now())
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  role      Role      @relation(fields: [roleId], references: [id], onDelete: Cascade)

  @@unique([userId, roleId])
  @@index([userId])
  @@index([roleId])
  @@map("user_roles")
}
```

#### 2.1.3 `permissions` - 权限表

```prisma
model Permission {
  id          String    @id @default(cuid())
  code        String    @unique  // 权限编码，如 "order:create"
  name        String    // 权限名称，如 "创建订单"
  description String?
  module      String    // 所属模块，如 "order", "inventory"
  action      String    // 操作类型，如 "create", "read", "update", "delete"
  isSystem    Boolean   @default(true)
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
  roles       Role[]

  @@index([module])
  @@index([action])
  @@map("permissions")
}
```

## 3. API 设计

### 3.1 RBAC 权限管理 API

#### 3.1.1 角色管理

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | `/api/roles` | 获取角色列表 | `role:read` |
| GET | `/api/roles/:id` | 获取角色详情 | `role:read` |
| POST | `/api/roles` | 创建角色 | `role:create` |
| PUT | `/api/roles/:id` | 更新角色 | `role:update` |
| DELETE | `/api/roles/:id` | 删除角色 | `role:delete` |
| GET | `/api/roles/:id/permissions` | 获取角色权限 | `role:read` |
| PUT | `/api/roles/:id/permissions` | 更新角色权限 | `role:update` |

**请求示例（创建角色）：**
```json
POST /api/roles
{
  "name": "销售经理",
  "code": "sales_manager",
  "description": "管理销售团队",
  "permissions": ["order:read", "order:create", "order:update", "customer:read"]
}
```

**响应示例：**
```json
{
  "id": "role_123",
  "name": "销售经理",
  "code": "sales_manager",
  "description": "管理销售团队",
  "permissions": ["order:read", "order:create", "order:update", "customer:read"],
  "createdAt": "2026-03-25T00:00:00Z"
}
```

#### 3.1.2 用户角色分配

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | `/api/users/:id/roles` | 获取用户角色 | `user:read` |
| POST | `/api/users/:id/roles` | 给用户分配角色 | `user:update` |
| DELETE | `/api/users/:id/roles/:roleId` | 移除用户角色 | `user:update` |

#### 3.1.3 权限管理

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | `/api/permissions` | 获取所有权限列表 | `permission:read` |
| GET | `/api/permissions/modules` | 获取按模块分组的权限 | `permission:read` |
| GET | `/api/my/permissions` | 获取当前用户权限 | 不需要特殊权限 |

## 4. 权限中间件设计

### 4.1 整体架构

```
请求进入
    ↓
JWT 认证 → 解析用户 ID
    ↓
检查用户是否有效
    ↓
加载用户所有角色 → 展开所有权限
    ↓
检查接口权限 → 是否有当前操作权限
    ↓ 权限检查通过
应用行级数据过滤 → 自动添加 ownerId/OrganizationId 过滤条件
    ↓
处理请求
    ↓
返回响应
```

### 4.2 权限检查流程

```typescript
// 伪代码
async function permissionMiddleware(req, res, next) {
  // 1. 从 JWT 获取用户 ID
  const userId = getUserIdFromToken(req);
  if (!userId) {
    return res.status(401).json({ error: '未认证' });
  }

  // 2. 加载用户信息和所有角色
  const user = await userService.findById(userId);
  const userRoles = await roleService.getUserRoles(userId);
  const permissions = permissionService.mergePermissions(userRoles);

  // 3. 检查当前路由是否需要特定权限
  const requiredPermission = getRequiredPermission(req.route);
  if (!permissions.includes(requiredPermission)) {
    return res.status(403).json({ error: '无权限' });
  }

  // 4. 将用户和权限信息存入请求上下文
  req.context = {
    userId,
    organizationId: user.organizationId,
    permissions,
    ownerId: userId  // 用于行级隔离
  };

  // 5. 继续处理
  next();
}
```

### 4.3 行级数据隔离

#### 4.3.1 隔离策略

我们采用**应用层行级过滤**方案，而非数据库 RLS：

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 数据库 RLS | 数据库层面强制，安全 | 性能开销，依赖 PostgreSQL 特性，调试复杂 | 不选 |
| 应用层自动过滤 | 性能好，易调试，跨数据库兼容 | 需要开发人员配合 | ✅ 选择 |

#### 4.3.2 实现方式

在服务层基类中，自动添加 `ownerId` 过滤条件：

```typescript
// base.service.ts
protected addRowLevelFilter<UserRowLevel = { ownerId?: string }>(
  where: Record<string, any>,
  context: RequestContext
): Record<string, any> {
  // 如果模型支持 ownerId 隔离，自动添加过滤
  if (this.supportsRowLevel && context.userId) {
    return {
      ...where,
      ownerId: context.userId
    };
  }
  // 如果支持 organizationId 隔离
  if (this.supportsOrgLevel && context.organizationId) {
    return {
      ...where,
      organizationId: context.organizationId,
      ...where
    };
  }
  return where;
}
```

使用示例：

```typescript
// customer.service.ts
async findAll(where: WhereInput, context: RequestContext) {
  // 自动添加 ownerId 过滤
  const filteredWhere = this.addRowLevelFilter(where, context);
  return this.prisma.customer.findMany({
    where: filteredWhere
  });
}
```

#### 4.3.3 支持行级隔离的表

| 表 | 隔离字段 | 隔离级别 |
|----|---------|---------|
| customers | `ownerId` | 用户级别 |
| suppliers | `ownerId` | 用户级别 |
| orders | `salesRepId` | 用户级别 |
| products | 暂无 | 计划组织级别 |

### 5.5 Webhook 接收

阿里国际站可以配置订单状态变更推送：

```
回调 URL: https://your-domain.com/api/platforms/webhook/alibaba
```

处理流程：
1. 验证签名（防止非法请求）
2. 获取事件类型和订单 ID
3. 异步触发该订单的同步
4. 返回 200 确认接收

### 5.6 幂等处理

为了避免重复处理，采用以下策略：

1. **唯一约束**：`(platformId, platformOrderNo)` 唯一约束
2. **存在判断**：同步前先查询是否存在
3. **变更检测**：只有当平台更新时间晚于本地更新时间才更新
4. **幂等键**：webhook 事件使用事件 ID 作为幂等键，已处理事件直接跳过

### 5.7 错误处理与重试

| 错误类型 | 处理方式 |
|----------|---------|
| API 限流 | 指数退避重试，最大 3 次 |
| 令牌过期 | 自动刷新令牌后重试 |
| 网络错误 | 重试 3 次，失败记录日志 |
| 数据格式错误 | 跳过该订单，记录错误信息，继续处理 |
| 数据库错误 | 终止同步，回滚，记录错误 |

## 8. 安全考虑

1. **平台密钥加密存储**：所有 `apiKey`, `apiSecret`, `accessToken` 加密存储在数据库，使用应用层加密
2. **Webhook 签名验证**：所有平台回调必须验证签名，防止非法请求
3. **权限最小化**：RBAC 遵循最小权限原则
4. **行级隔离**：用户只能看到自己的数据
5. **日志审计**：所有同步和发布操作都有日志记录

---

## 9. 性能考虑

1. **分页拉取**：订单同步分页拉取，避免一次性拉取过多数据
2. **异步处理**：webhook 接收后异步处理，立即返回
3. **批量操作**：批量更新库存，减少数据库事务次数
4. **索引优化**：添加适当的联合索引，加速查询
5. **缓存**：权限列表缓存到 Redis，减少数据库查询
