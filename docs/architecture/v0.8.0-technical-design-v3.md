```
Accounts ──> Transactions
```

### 2.2 关键设计要点

**1. 加密存储敏感信息**
- `clientSecret`, `accessToken`, `refreshToken`, `apiSecret` 等敏感信息需要加密存储
- 使用 AES-256-GCM 加密，密钥从环境变量读取

**2. 唯一约束设计**
- `platformOrderId + platformAccountId` 唯一约束：防止同一平台账号重复同步同一订单
- `ruleId + userId + period` 唯一约束：防止重复计算绩效

**3. 多币种支持**
- 所有金额字段同时保存原币和人民币（或本位币）
- `exchangeRate` 保存 6 位小数精度保证汇兑准确

**4. 索引优化**
- 外键必加索引
- 时间范围查询常用字段（`createdAt`, `orderDate`, `transactionDate`）建立索引
- 状态+时间复合索引加快筛选

## 3. 认证授权架构

### 3.1 整体架构: JWT + RBAC

```
┌─────────────┐
│  User Login │
└──────┬──────┘
       │
       ▼
┌────────────────────┐   ┌───────────┐
│ Generate JWT Token │──▶│  Redis    │  Store token blacklist
└─────────────┬──────┘   └───────────┘
              │
              ▼
┌────────────────────────┐
│  Request with Token    │
│  Authorization: Bearer │
└─────────────┬──────────┘
              │
              ▼
┌────────────────────────┐  Yes  ┌──────────────────┐
│ Token Expired?         │──────▶│  401 Unauthorized │
└─────────────┬──────────┘        └──────────────────┘
              │ No
              ▼
┌────────────────────────┐  Yes  ┌──────────────────┐
│ Token in Blacklist?    │──────▶│  401 Unauthorized │
└─────────────┬──────────┘        └──────────────────┘
              │ No
              ▼
┌───────────────────────────────────┐
│ Load User + User Permissions      │
└─────────────┬─────────────────────┘
              │
              ▼
┌───────────────────────────────────┐  Yes  ┌──────────────────┐
│ Check Permission (middleware)     │──────▶│  403 Forbidden   │
└─────────────┬─────────────────────┘        └──────────────────┘
              │ No
              ▼
┌────────────────────────┐
│  Proceed to API Route  │
└────────────────────────┘
```

### 3.2 RBAC 数据模型

```
用户 (User)
  ↓ 多对多
用户角色 (UserRole)
  ↓ 多对多
角色 (Role)
  ↓ 多对多
角色权限 (RolePermission)
  ↓ 多对多
权限 (Permission)
```

**权限粒度设计:**
- 模块级: `order:*` 表示所有订单权限
- 操作级: `order:create`, `order:read`, `order:update`, `order:delete`, `order:approve`
- 数据级: 通过业务员关联实现数据隔离

### 3.3 权限缓存策略

```typescript
// 权限缓存到 Redis，key: `permissions:userId`
// 角色变更时自动失效缓存
// TTL: 24 小时
```

### 3.4 API 权限中间件

```typescript
// 用法:
// requirePermission('order:approve')
// requirePermission('order:*')

function requirePermission(permission: string) {
  return (req, res, next) => {
    const user = req.user;
    if (!user.hasPermission(permission)) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  };
}
```

### 4.1 架构设计: 适配器模式

```
┌─────────────────────────────────────────────────────────────┐
│                     PlatformService                          │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────┐ │
│  │  Alibaba   │ │  Amazon    │ │  TikTok    │ │ Shopify │ │
│  │  Adapter   │ │  Adapter   │ │  Adapter   │ │ Adapter │ │
│  └─────────────┘ └─────────────┘ └─────────────┘ └─────────┘ │
│           │              │              │              │      │
│           └──────────────┴──────────────┴──────────────┘      │
│                    ↓                          │                │
│          ┌───────────────────────────────────┘                │
│          │ 统一接口: syncOrders(), syncInventory()           │
│          │ 统一数据格式转换到 PlatformOrder                  │
└──────────┼──────────────────────────────────────────────────┘
           │
           ▼
┌──────────────────────────────────────────────────┐
│         同步流程                                  │
│                                                  │
│  1. 获取增量订单 (从 lastSyncAt 开始)             │
│  2. 检查平台订单ID是否已存在                       │
│  3. 不存在 → 新建，存在 → 对比更新                   │
│  4. 转换商品信息，关联内部 SKU                     │
│  5. 更新 lastSyncAt                               │
│  6. 触发利润核算队列（异步）                       │
└──────────────────────────────────────────────────┘
```

### 4.2 支持平台

| 平台 | 认证方式 | API 权限 | 同步内容 |
|------|---------|---------|---------|
| **阿里国际站** | OAuth 2.0 | 订单、产品 | 订单全量同步 |
| **亚马逊** | OAuth LWA | 订单、库存、财务 | 订单 + FBA 库存 |
| **TikTok Shop** | OAuth 2.0 | 订单、产品 | 订单同步 |
| **Shopify** | API Key + Secret | Webhook + API | 订单实时同步 |

### 4.3 同步策略

**增量同步**
- 定时拉取（间隔 15 分钟 - 1 小时可配置）
- 只拉取更新时间 > lastSyncAt 的订单
- 减少 API 调用，提高效率

**幂等处理**
- 以平台订单ID + 账号ID作为唯一键
- 重复同步只更新不新增
- 支持 webhook 实时推送和定时拉取双活

**错误处理**
- 记录错误信息到 `errorMessage`
- 更新 `syncStatus` 为 FAILED
- 支持手动重试
- 告警通知（配置项）

### 4.4 数据映射

| 平台字段 → ERP 字段 | 说明 |
|-------------------|------|
| platform_order_id → platformOrderId | 保留原始ID |
| total_amount → totalAmount | 按币种转换 |
| line_items → items JSON | 保留原始明细 |
| buyer → buyerName/buyerEmail | 提取买家信息 |
| shipping_address → shippingAddress JSON | 保存完整地址 |

## 5. 海外仓同步机制

### 5.1 整体架构

```
┌──────────────────────┐
│  亚马逊 FBA          │
│  第三方海外仓 API    │
└──────────┬───────────┘
           │  API 调用
           ▼
┌──────────────────────┐
│  海外仓库存同步服务    │
│  1. 获取全量库存      │
│  2. 对比本地库存      │
│  3. 更新差异          │
│  4. 记录同步时间      │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  OverseasInventory   │
│  (本地存储)           │
└──────────────────────┘
```

### 5.2 库存模型

```prisma
model OverseasInventory {
  // 核心字段:
  warehouseId       // 关联海外仓
  productId         // 关联产品
  sku               // SKU
  fnsku             // FBA SKU
  asin              // ASIN
  availableQty      // 可售数量
  inboundQty        // 在途数量
  reservedQty       // 预留数量
  totalQty          // 总数量 = 可售 + 在途 + 预留
  costPerUnit       // 单位成本（用于利润计算）
}
```

### 5.3 FBA 同步流程

1. **获取 FBA 库存** 调用亚马逊 Inventory API
2. **解析响应** 提取每个 SKU 的可售数量
3. **匹配产品** 通过 fnsku/asin/sku 匹配内部产品
4. **更新库存** 覆盖或增量更新
5. **记录日志** 记录同步结果

**同步频率:** 每日 1-4 次可配置（根据 API 配额）

### 5.4 补货建议算法（预留扩展）

```
建议补货数量 = (最大库存天数 - 当前可售天数) × 日均销量

其中:
- 当前可售天数 = availableQty / 日均销量
- 日均销量 = 最近 30 天销量 / 30
- 最大库存天数 = 配置参数（如 45 天）
```

## 7. 财务模块架构

### 7.1 核心模型

```
Account (账户)
  └─> Transaction (交易流水)

Expense (费用)
  └─> Reimbursement (报销单)
    └─> Transaction (付款交易)
```

### 7.2 记账规则

**复式记账简化版本（适合中小企业）**
- 每笔交易记录一个账户的增减
- 转账操作记录两笔交易（转出 + 转入）
- 保持账户余额实时一致

**余额更新:**
```typescript
// 创建交易后，自动更新账户余额
account.currentBalance += (type === INCOME || type === TRANSFER_IN)
  ? amountCny : -amountCny;
await account.save();
```

### 7.3 费用分类

预置标准分类:
- **固定费用:** 租金、水电、工资、社保
- **运营费用:** 营销、广告、物流、仓储、包装
- **平台费用:** 佣金、手续费、广告费
- **管理费用:** 差旅、招待、办公、软件
- **财务费用:** 手续费、利息

### 7.4 报销流程

1. 员工提交报销单，上传发票凭证
2. 部门经理审批（判断费用合规性）
3. 财务审批（核对发票）
4. 批准后从指定账户打款
5. 生成交易流水，自动记账

## 8. 利润核算引擎设计

### 8.1 核算维度

利润核算支持多维切片:
- ✅ 按订单: 每张订单利润明细
- ✅ 按 SKU: 每个产品利润统计
- ✅ 按平台: 每个平台/店铺利润统计
- ✅ 按人员: 每个业务员利润贡献
- ✅ 按时间段: 日/周/月/季度/年

### 8.2 利润计算公式

**单订单利润计算:**

```
销售收入 = 商品售价 × 数量 + 运费收入
─────────────────────────────────────────
产品成本 = 采购成本 × 数量
国内运费 = 物流到仓运费
国际运费 = 头程运费
仓储费 = 按数量分摊的海外仓费用
包装费 = 包装材料成本
─────────────────────────────────────────
平台佣金 = 销售收入 × 佣金率
支付手续费 = 销售收入 × 手续费率
广告费用 = 分摊广告成本
税费 = 关税/增值税等
─────────────────────────────────────────
其他费用 = 其他分摊费用
─────────────────────────────────────────
总成本 = 产品成本 + 国内运费 + 国际运费 + 仓储费 +
         包装费 + 平台佣金 + 支付手续费 + 广告费用 +
         税费 + 其他费用

毛利润 = 销售收入 - 总成本
净利润 = 毛利润 - 固定费用分摊

利润率 = (净利润 / 销售收入) × 100%
```

### 8.3 分摊算法

**广告费用分摊:**
- 按 SKU 销售额占比分摊到订单/商品

**固定费用分摊:**
- 房租/水电/工资等固定费用
- 按销售收入占比分摊到本期订单

**仓储费用分摊:**
- 按库存天数 × 数量 分摊

### 8.4 异步处理架构

```
用户触发利润核算 → 创建 ProfitCalculation (状态 DRAFT)
                 ↓
         放入后台任务队列
                 ↓
         工作进程异步处理:
           1. 查询时间范围内所有订单
           2. 逐单计算利润明细
           3. 插入 ProfitDetail
           4. 汇总统计到 ProfitCalculation
           5. 更新状态 = COMPLETED
                 ↓
         WebSocket 推送完成通知
```

**优势:**
- 用户界面不阻塞
- 支持大数据量核算
- 失败可重试

### 8.5 存储设计

- `profit_calculations`: 核算汇总，存储总计数据
- `profit_details`: 明细按订单存储，支持钻取查询
- 保留历史版本，可对比不同时期核算结果

## 9. 绩效提成计算引擎

### 9.1 规则驱动设计

规则定义 → 应用到业务员周期业绩 → 计算提成

### 9.2 支持的提成规则

| 规则类型 | 计算方式 | 适用场景 |
|---------|---------|---------|
| **销售额提成** | 销售额 × 提成比例 | 按销售额提成 |
| **毛利提成** | 毛利 × 提成比例 | 鼓励高毛利产品 |
| **净利润提成** | 净利润 × 提成比例 | 最精准，鼓励净利 |
| **订单数量** | 固定金额每单 | 新客户开发 |
| **新客户** | 固定奖金每个新客户 | 开发奖励 |
| **目标达成** | 达标额外奖金 | 目标管理 |
| **阶梯提成** | 不同区间不同比例 | 多劳多得 |

**阶梯提成示例配置:**
```json
[
  { "min": 0, "max": 10000, "rate": 0.02 },
  { "min": 10000, "max": 30000, "rate": 0.03 },
  { "min": 30000, "rate": 0.05 }
]
```
计算: 销售额 25000 → 10000×2% + 15000×3% = 200 + 450 = 650

### 9.3 计算流程

```
1. 确定结算周期 (月份)
2. 获取该周期内业务员负责的所有订单利润
3. 汇总业绩指标:
   - 总销售额
   - 总毛利
   - 总净利
   - 订单数量
   - 新客户数
4. 应用绩效规则计算各项提成
5. 累加得到总提成
6. 创建佣金单，等待审批
7. 审批通过后付款，生成财务交易
```

### 9.4 数据模型

- `performance_rules`: 规则定义，支持多规则
- `performance_metrics`: 按人员/周期统计业绩指标
- `commissions`: 提成结算单，关联业绩明细

### 9.5 业绩数据来源

业绩数据直接来自利润核算的 `profit_details`，保证数据一致性:
- `salesRepId` 关联业务员
- `netProfit` 直接拿来计算提成

## 12. 安全考虑

### 12.1 敏感数据加密

- 平台 API 密钥、token 等敏感信息加密存储
- 加密密钥来自环境变量，不提交到代码库
- 数据库连接密码使用 Docker secrets 或环境变量

### 12.2 权限控制

- 数据级权限: 业务员只能看到自己的订单
- 管理员可查看所有数据
- 审批权限必须经过流程，不能绕过
- 操作日志可追溯

### 12.3 API 安全

- CORS 限制允许来源
- JWT token 过期时间: 24 小时
- 登出即加入黑名单
- 频率限制防止暴力破解

### 12.4 审计日志

- 敏感操作（删除、修改财务数据）记录操作人、时间、IP
- 审批操作完整历史不可篡改

## 13. 性能考虑

### 13.1 数据库优化

- 正确建立索引（见 schema 设计）
- 大表分页查询，避免全表扫描
- 定期清理无用数据（可归档到历史表）

### 13.2 缓存策略

- 权限信息缓存到 Redis
- 不常变化的基础数据（产品、客户）缓存
- 核算结果缓存，重复查询直接返回

### 13.3 异步处理

- 订单同步、利润核算、提成计算都走异步队列
- 用户界面快速响应
- 后台 worker 处理耗时任务

### 13.4 批量操作

- 批量插入核算明细，减少事务次数
- 批量更新库存，提高效率

## 14. 扩展性考虑

### 14.1 新增平台支持

新增平台只需要:
1. 新增平台适配器实现 `PlatformAdapter` 接口

### 14.3 新增提成规则

新增提成计算方式只需要:
1. 添加新的规则类型到枚举
2. 实现新的计算逻辑
3. 前端配置界面支持即可

- 支持更多平台: 1688, eBay, Wish 等
