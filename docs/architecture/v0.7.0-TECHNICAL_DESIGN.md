## 2. 架构概述

### 2.1 项目背景

Trade ERP 是面向外贸企业的综合企业资源管理系统，涵盖以下核心业务模块：

- **客户管理**: 客户信息、联系人管理
- **询盘报价**: 询盘处理、报价单管理
- **销售订单**: 订单全生命周期管理、生产跟踪、质检
- **采购管理**: 供应商管理、采购订单、入库、付款
- **产品研发**: 产品分类、产品调研、竞品对比
- **仓储管理**: 多仓库管理、库存管理、出入库
- **财务报表**: 利润分析、销售统计、库存报表
- **系统认证**: 用户认证、权限管理

5. **测试可测**: 服务层独立，便于单元测试
6. **向后兼容**: API 变更保持兼容性

### 4.2 已发现问题与优化建议

#### 问题 1: `Inventory` vs `InventoryItem` 表冗余

**现状:**
- `InventoryItem`: `unique([productId, warehouse])`
- `Inventory`: `unique([productId, warehouseId])`

两个表功能重复，都是记录产品在仓库的库存数量。

**优化方案:** ✅ **已在 v0.7.0 统一**保留 `InventoryItem`，移除 `Inventory`。推荐使用:

```prisma
model InventoryItem {
  id            String    @id @default(cuid())
  productId     String
  warehouse     String    @default("MAIN")  // 使用仓库 code 而非外键
  quantity      Int       @default(0)
  reservedQty   Int       @default(0)
  availableQty  Int       @default(0)
  location      String?
  lastCountedAt DateTime?
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
  product       Product   @relation(fields: [productId], references: [id])

  @@unique([productId, warehouse])
  @@map("inventory_items")
}
```

**原因:**
- `Warehouse` 是基础数据，不会频繁变动
- 使用 `warehouse` code 直接存储比 join 性能更好
- 减少一次 join 查询，提升查询性能

#### 问题 2: 索引优化

**现状:** 大部分关键字段已有索引，但仍有可以优化的地方。

**新增索引建议:**

```prisma
// StockMovement
@@index([productId, createdAt])
@@index([type, createdAt])

// InventoryLog
@@index([productId, warehouseId])
// ✅ 已有

// OutboundOrder
@@index([warehouseId, status])
// ✅ 已有

// Payment
@@index([orderId, paymentDate])
// ✅ 已有

// SupplierPayment
@@index([purchaseOrderId, paymentDate])
// ✅ 已有
```

**结论:** 当前索引设计基本合理，无需大规模调整。

#### 问题 3: 缺少库存盘点模型

**建议新增:** 库存盘点单模型，支持周期盘点。

```prisma
model StockCount {
  id              String          @id @default(cuid())
  countNo         String          @unique
  warehouseId     String
  status          StockCountStatus @default(DRAFT)
  countDate       DateTime?
  countedBy       String?
  totalItems      Int             @default(0)
  countedItems    Int             @default(0)
  differenceQty   Int             @default(0)
  differenceAmount Decimal        @default(0) @db.Decimal(12, 2)
  notes           String?
  createdAt       DateTime        @default(now())
  updatedAt       DateTime        @updatedAt
  items           StockCountItem[]
  warehouse       Warehouse       @relation(fields: [warehouseId], references: [id])

  @@index([warehouseId, status])
  @@index([countDate])
  @@map("stock_counts")
}

model StockCountItem {
  id              String          @id @default(cuid())
  stockCountId    String
  productId       String
  systemQty       Int
  actualQty       Int
  difference      Int
  differenceReason String?
  notes           String?
  createdAt       DateTime        @default(now())
  stockCount      StockCount      @relation(fields: [stockCountId], references: [id], onDelete: Cascade)
  product         Product         @relation(fields: [productId], references: [id])

  @@index([stockCountId])
  @@index([productId])
  @@map("stock_count_items")
}

enum StockCountStatus {
  DRAFT
  IN_PROGRESS
  COMPLETED
  CANCELLED
}
```

**优先级:** 低，v0.8.0 实现。

#### 问题 4: 缺少批次管理支持

当前设计中，`InboundOrderItem` 已有 `batchNo` 字段，但库存层面不按批次分开存储。对于需要批次追踪的行业（如食品、化妆品）需要支持。

**建议:** 在 `InventoryItem` 中增加 `batchNo` 作为唯一约束的一部分:

```prisma
// 未来版本: 批次级库存管理
@@unique([productId, warehouse, batchNo])
```

**优先级:** 低，根据业务需求决定是否实现。当前设计已预留字段，扩展方便。

## 6. 仓储管理模块设计

### 6.1 业务流程

```
采购入库:
PurchaseOrder → InboundOrder → Confirm → 增加库存 → 生成 StockMovement → 生成 InventoryLog

销售出库:
Order → OutboundOrder → Confirm → 扣减库存 → 生成 StockMovement → 生成 InventoryLog

库存调拨:
InboundOrder (transfer in) + OutboundOrder (transfer out)

库存调整:
Inbound/Outbound with type ADJUSTMENT
```

### 6.2 核心 API - 入库管理

#### 6.2.1 获取入库单列表

```http
GET /api/v1/inbound-orders
```

**查询参数:** 支持标准分页 + 筛选:
- `type`: 入库类型
- `status`: 状态筛选
- `warehouseId`: 仓库筛选
- `supplierId`: 供应商筛选
- `purchaseOrderId`: 采购单筛选

## 8. 安全与认证架构

- **Token 过期**: 24 小时

### 8.2 权限模型

## 9. 性能优化方案

### 9.1 数据库查询优化

1. **正确使用索引**: 外键、常用筛选字段必须建索引
2. **避免 N+1**: 使用 `include` 一次性加载关联数据
3. **分页**: 所有列表查询必须分页，默认 20 条每页
4. **计数优化**: 复杂统计使用 Materialized View (PostgreSQL)

### 9.2 缓存策略

| 缓存场景 | 缓存方式 | TTL |
|----------|----------|-----|
| 产品基本信息 | 内存缓存 | 5 分钟 |
| 报表数据 | 数据库存储 | 每天刷新 |
| 用户会话 | JWT | 24 小时 |
| 仪表板数据 | 内存缓存 | 1 分钟 |

### 9.3 前端优化

1. **代码分割**: Next.js 自动按路由分割
2. **图片优化**: Next.js Image 组件自动优化
3. **Server Components**: 减少客户端 JavaScript

### 9.4 报表查询优化

大型报表生成策略:

1. **异步生成**: 报表生成放入后台队列，完成后通知
2. **预生成**: 常用日报/周报/月报提前生成好
3. **汇总表**: 日汇总表预先计算，查询直接读取
