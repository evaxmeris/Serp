### 3.8 库存仓储模块

**模型:** `Warehouse`, `Inventory`, `InventoryItem`, `InventoryLog`, `StockMovement`, `InboundOrder`, `InboundOrderItem`, `OutboundOrder`, `OutboundOrderItem`

**审查结果:** ⚠️ **存在冗余，需要优化**

**问题分析:**

1. **`Inventory` vs `InventoryItem` 功能重复:**

   - `Inventory`: `@@unique([productId, warehouseId])` - 使用仓库 ID 外键
   - `InventoryItem`: `@@unique([productId, warehouse])` - 使用仓库 code

   两个表都是记录产品在仓库的库存，功能完全重复。

2. **设计对比:**

   | 维度 | Inventory | InventoryItem | 推荐 |
   |------|-----------|---------------|------|
   | 外键关联 | 是 (warehouseId → Warehouse.id) | 否 (直接存 code) | InventoryItem |
   | 查询性能 | 需要 join Warehouse | 不需要 join | InventoryItem 胜 |
   | 一致性 | 如果删除仓库，数据需要处理 | code 基本不变，稳定 | InventoryItem 胜 |
   | 简洁性 | 复杂 | 简单 | InventoryItem 胜 |

**结论:** 保留 `InventoryItem`，移除 `Inventory`。`InventoryLog` 需要调整外键关系。

3. **其他检查:**

- `Warehouse` 模型设计正确 ✅
- `InventoryLog` 完整记录每一笔库存变动，可追溯 ✅
- `StockMovement` 和 `InventoryLog` 功能略有重复，但粒度不同。`StockMovement` 是汇总变动，`InventoryLog` 是明细。保留两者 ✅
- `InboundOrder`/`OutboundOrder` 设计正确，支持不同类型出入库 ✅
- 索引设计完整 ✅

## 4. 问题发现

| 优先级 | 问题 | 模块 | 影响范围 |
|--------|------|------|----------|
| 高 | `Inventory` 与 `InventoryItem` 表冗余 | 库存 | 所有库存操作 |
| 中 | `Customer` 缺失常用索引 | 客户 | 列表查询 |
| 中 | `Inquiry` 缺失按负责人索引 | 询盘 | 任务分配查询 |
| 低 | 缺少库存盘点模型 | 库存 | 盘点流程 |
| 低 | 缺少批次级库存支持 | 库存 | 批次追踪需求 |

### 4.1 高优先级问题: 表冗余

**现状:**
```prisma
// 表 1: Inventory - 使用 warehouseId 外键
model Inventory {
  id                String         @id @default(cuid())
  productId         String
  warehouseId       String
  quantity          Int            @default(0)
  availableQuantity Int            @default(0)
  lockedQuantity    Int            @default(0)
  ...
  @@unique([productId, warehouseId])
}

// 表 2: InventoryItem - 使用 warehouse code
model InventoryItem {
  id            String    @id @default(cuid())
  productId     String
  warehouse     String    @default("MAIN")
  quantity      Int       @default(0)
  reservedQty   Int       @default(0)
  availableQty  Int       @default(0)
  ...
  @@unique([productId, warehouse])
}
```

**问题:**
- 两张表存储相同信息（产品在仓库的库存）
- 应用代码需要维护两份数据，容易不一致
- 占用多余存储空间

**解决方案:** 移除 `Inventory`，统一使用 `InventoryItem`。

**影响范围:**
- 需要修改 `InventoryLog` 关联关系
- 需要修改所有读写库存的服务代码
- 需要数据迁移

### 4.2 中优先级问题: 缺失索引

**缺失索引 1 - Customer:**
```prisma
// 建议添加:
@@index([status, createdAt])
@@index([ownerId])
```

**缺失索引 2 - Inquiry:**
```prisma
// 建议添加:
@@index([assignedTo, status])
```

**影响:** 查询时全表扫描，数据量大后性能下降。

### 4.3 低优先级问题: 功能缺失

**问题 1: 缺少库存盘点模型**

当前没有盘点单模型，无法支持周期盘点流程。建议新增 `StockCount` 和 `StockCountItem`。

**问题 2: 缺少批次级库存支持**

当前库存按产品+仓库汇总，不按批次分开。对于需要批次追踪的行业（食品、化妆品、医药）不支持。

**当前设计已预留 `batchNo` 在 `InboundOrderItem` 中，扩展方便。**

## 5. 优化建议

### 5.1 高优先级优化: 移除冗余表

**执行步骤:**

1. **数据迁移准备:**
   - 将现有 `Inventory` 数据迁移到 `InventoryItem`
   - `Inventory.warehouseId` → 找到对应 `Warehouse.code` → `InventoryItem.warehouse`
   - `Inventory.quantity` → `InventoryItem.quantity`
   - `Inventory.availableQuantity` → `InventoryItem.availableQty`
   - `Inventory.lockedQuantity` → `InventoryItem.reservedQty`

2. **修改 `InventoryLog` 关系:**

```prisma
// 修改前:
model InventoryLog {
  ...
  inventoryId     String?
  inventory       Inventory? @relation(fields: [inventoryId], references: [id])
}

// 修改后:
model InventoryLog {
  ...
  // 移除 inventoryId 外键
  // productId + warehouseId 已经存在，足够定位
}
```

3. **删除 `Inventory` 模型**

4. **更新所有服务代码:**
   - 搜索所有引用 `inventory` 的代码
   - 替换为 `inventoryItem`
   - 测试所有库存操作

5. **验证数据一致性**

### 5.3 低优先级优化: 新增库存盘点模型 (v0.8.0)

```prisma
model StockCount {
  id              String          @id @default(cuid())
  countNo         String          @unique
  warehouseId     String
  status          StockCountStatus @default(DRAFT)
  countDate       DateTime?
  countedBy       String?
  totalItems      Int             @default(0)
  countedItems    Int             @default(0)
  differenceQty   Int             @default(0)
  differenceAmount Decimal        @default(0) @db.Decimal(12, 2)
  notes           String?
  createdAt       DateTime        @default(now())
  updatedAt       DateTime        @updatedAt
  items           StockCountItem[]
  warehouse       Warehouse       @relation(fields: [warehouseId], references: [id])

  @@index([warehouseId, status])
  @@index([countDate])
  @@map("stock_counts")
}

model StockCountItem {
  id              String          @id @default(cuid())
  stockCountId    String
  productId       String
  systemQty       Int
  actualQty       Int
  difference      Int
  differenceReason String?
  notes           String?
  createdAt       DateTime        @default(now())
  stockCount      StockCount      @relation(fields: [stockCountId], references: [id], onDelete: Cascade)
  product         Product         @relation(fields: [productId], references: [id])

  @@index([stockCountId])
  @@index([productId])
  @@map("stock_count_items")
}

enum StockCountStatus {
  DRAFT
  IN_PROGRESS
  COMPLETED
  CANCELLED
}
```

## 6. 迁移计划

### 6.1 迁移类型

这是**破坏性变更**，需要停服迁移。

### 6.2 迁移步骤

```bash
# 1. 备份数据库
pg_dump -U $user $db > backup-v0.6.0-$(date +%Y%m%d).sql

# 2. 创建迁移文件
npx prisma migrate dev --name v0.7.0-remove-redundant-inventory-table

# 3. 编写数据迁移脚本
# scripts/migrate-inventory-to-inventory-item.ts

# 4. 执行迁移
npx prisma migrate deploy

# 5. 执行数据迁移脚本
ts-node scripts/migrate-inventory-to-inventory-item.ts

# 6. 验证数据
# 比较计数: SELECT COUNT(*) FROM inventory; vs SELECT COUNT(*) FROM inventory_items;
# 抽查几个产品的库存数量是否一致

# 7. 测试核心功能
# - 入库操作
# - 出库操作
# - 库存查询
# - 库存变动记录

# 8. 确认无误后删除原表
# (由下次 migration 处理)
```

### 6.3 回滚方案

```bash
# 1. 恢复备份
psql -U $user $db < backup-v0.6.0-xxxxxx.sql

# 2. 回滚迁移
npx prisma migrate resolve --rolled-back v0.7.0-remove-redundant-inventory-table

# 3. 重启应用
```
