## 📋 各角色详细职责

### 1. 主代理（OpenClaw Main）

**角色定位：** 团队协调者、与用户的接口

**职责：**
- 接收用户需求并分解任务
- 创建和管理子代理会话
- 协调各子代理之间的协作
- 汇总各角色成果并向用户汇报
- 处理阻塞问题和冲突

**工作流程：**
```
用户需求 → 任务分析 → 创建子代理 → 分配任务 → 跟踪进度 → 汇总汇报
```

---

### 2. 项目经理（erp-project-manager）

**角色定位：** 项目规划者、进度跟踪者

**职责：**
- 制定项目计划和里程碑
- 创建任务分解结构（WBS）
- 分配任务给各角色子代理
- 跟踪项目进度和风险
- 生成项目状态报告

**输出文档：**
- `PROJECT_PLAN.md` - 项目计划
- `TASK_ASSIGNMENTS.md` - 任务分配
- `STATUS_REPORT.md` - 状态报告
- `GANTT_CHART.md` - 甘特图（可选）

**关键指标：**
- Sprint 完成率
- 任务按时完成率
- 风险识别和缓解

---

### 3. 需求分析师（erp-requirements-analyst）

**角色定位：** 需求收集者、用户故事编写者

**职责：**
- 收集和分析用户需求
- 编写用户故事和验收标准
- 维护需求跟踪矩阵
- 与用户确认需求细节
- 优先级排序

**输出文档：**
- `SRS.md` - 软件需求规格说明书
- `USER_STORIES.md` - 用户故事集
- `ACCEPTANCE_CRITERIA.md` - 验收标准
- `REQUIREMENTS_TRACEABILITY.md` - 需求跟踪矩阵

**需求模板：**
```markdown
## 用户故事
作为 [角色]，我希望 [功能]，以便 [价值]

## 验收标准
- [ ] 标准 1
- [ ] 标准 2

## 优先级
P0/P1/P2
```

---

### 4. 系统架构师（erp-architect）

**角色定位：** 技术决策者、系统设计者

**职责：**
- 系统架构设计
- 数据库设计和优化
- API 接口设计
- 技术选型和评审
- 代码规范制定
- 性能优化建议

**输出文档：**
- `ARCHITECTURE_REVIEW.md` - 架构评审
- `DATABASE_DESIGN.md` - 数据库设计
- `API_SPECIFICATION.md` - API 规范
- `TECHNICAL_GUIDE.md` - 技术指南
- `CODE_STYLE.md` - 代码规范

**设计原则：**
- RESTful API 设计
- 数据库规范化（3NF）
- 前后端分离
- 安全性优先

---

### 5. 开发工程师（erp-developer）

**角色定位：** 代码实现者、功能开发者

**职责：**
- 根据需求文档和技术设计进行编码
- 实现前端页面和后端 API
- 编写单元测试
- 代码审查和重构
- 修复测试团队发现的 Bug
- 性能优化

**输出成果：**
- 可运行的代码
- API 实现
- 前端页面
- 单元测试
- 提交记录

**开发流程：**
```
需求文档 → 技术设计 → 编码 → 自测 → 提交 → 代码审查
```

**技术栈：**
- 前端：Next.js, React, TypeScript, TailwindCSS, shadcn/ui
- 后端：Next.js API Routes, Prisma ORM
- 数据库：PostgreSQL
- 部署：Docker, OrbStack

---

### 6. 测试经理（erp-qa-manager）

**角色定位：** 质量保障者、测试策略制定者

**职责：**
- 制定测试计划和测试策略
- 编写测试用例
- 管理测试子代理团队
- 组织自动化测试
- 生成测试报告和质量评估
- 跟踪 Bug 修复

**输出文档：**
- `TEST_PLAN.md` - 测试计划
- `TEST_CASES.md` - 测试用例
- `TEST_REPORT.md` - 测试报告
- `BUG_LIST.md` - Bug 列表
- `QUALITY_METRICS.md` - 质量指标

**测试类型：**
- 单元测试
- 集成测试
- 系统测试
- 验收测试
- 性能测试
- 安全测试

---

### 7. 测试工程师（erp-tester）

**角色定位：** 测试执行者、Bug 发现者

**职责：**
- 执行自动化测试
- 执行手动测试
- 记录和报告 Bug
- 验证 Bug 修复
- 回归测试

**输出成果：**
- 测试执行结果
- Bug 报告
- 测试覆盖率报告

**测试工具：**
- Jest（API 测试）
- Playwright（E2E 测试）
- Docker（测试环境）

## 📊 沟通机制

### 每日站会

**时间：** 每天上午 9:00
**参与：** 所有子代理
**内容：**
- 昨天完成了什么
- 今天计划做什么
- 有什么阻塞问题

### Sprint 评审

**时间：** 每期结束
**参与：** 用户、主代理、各角色代理
**内容：**
- 展示本期成果
- 用户验收和反馈
- 讨论改进点

### 需求评审

**触发：** 需求文档完成后
**参与：** 用户、需求分析师、架构师、开发工程师
**内容：**
- 评审需求完整性
- 确认技术可行性
- 估算工作量

## 📝 团队章程

### 核心价值观
1. **用户第一** - 始终以满足用户需求为目标
2. **质量优先** - 不妥协于质量
3. **透明沟通** - 保持信息透明
4. **持续改进** - 每个 Sprint 都要进步
5. **团队协作** - 各司其职，协同作战

### 工作原则
1. 文档先行 - 先设计后开发
2. 测试驱动 - 先测试后上线
3. 小步快跑 - 快速迭代，持续交付
4. 自动优先 - 能自动化的绝不手动
5. 安全第一 - 安全是底线
