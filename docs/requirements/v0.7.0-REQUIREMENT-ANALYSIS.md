## 一、项目背景

### 1.1 项目概述

Trade ERP 是一款面向外贸企业的跨境电商 ERP 系统。当前版本 v0.6.0 存在**认证方案不统一**的问题：
- 同时使用 NextAuth v5 Beta 和自定义简化认证两种方案
- 认证状态可能不一致
- 代码维护复杂
- 存在安全隐患（缺少 CSRF 保护、速率限制等）

### 1.2 技术债务

| 问题 | 严重程度 | 影响 |
|------|----------|------|
| 认证方案不统一 | 🔴 高 | 维护困难，可能出现认证不一致 |
| 缺少 CSRF 保护 | 🟡 中 | 存在 CSRF 攻击风险 |
| 缺少登录速率限制 | 🔴 高 | 存在暴力破解风险 |
| NextAuth 配置不完整 | 🔴 高 | 部分功能可能失效 |

## 二、认证方案用户反馈分析

### 2.1 原有方案问题总结

通过对技术分析文档的梳理，总结用户/开发反馈如下：

#### 反馈 1: NextAuth v4 兼容性问题 ❌

**问题描述**:
- NextAuth v4 不兼容 Next.js 16 异步 Cookies API
- 出现 `cookies().getAll is not a function` 错误
- 动态路由 `params.nextauth` 参数解析失败

**用户反馈**:
> "登录后一直重定向到 csrf=true 页面，无法正常登录"
> "NextAuth 和 Next.js 16 不兼容，开发受阻"

**根因分析**: Next.js 16 破坏性变更，NextAuth v4 未跟进适配

---

#### 反馈 2: NextAuth v5 Beta 不稳定 ❌

**问题描述**:
- Beta 版本 API 可能变动
- `authorize` 回调未被正确调用
- Credentials 数据传递机制变更导致表单数据无法正确解析
- 文档不完善，调试困难

**用户反馈**:
> "按照文档配置了，但就是不调用 authorize 方法"
> "Beta 版坑太多，调试花了两天还是没搞定"

**根因分析**: Beta 版本功能不稳定，API 设计仍在迭代

---

#### 反馈 3: 双认证方案维护困难 ❌

**问题描述**:
- 项目同时存在两套认证逻辑
- API 路由一部分使用 NextAuth，一部分使用自定义认证
- 新手开发者难以理解
- 可能出现认证状态不同步

**用户反馈**:
> "看到两个认证库，不知道该用哪个"
> "改了这个地方，那个地方又出问题"

**根因分析**: 技术方案迭代过程中没有及时清理旧方案

---

#### 反馈 4: 缺少安全防护 ❌

**问题描述**:
- 登录接口无速率限制
- 可能被暴力破解
- 缺少密码强度验证
- 缺少登录审计日志

**用户反馈**:
> "安全检查说我们的登录接口有暴力破解风险"
> "需要符合基本的安全要求才能上线"

---

### 2.2 用户需求总结

| 需求点 | 优先级 | 说明 |
|--------|--------|------|
| 统一使用一种认证方案 | 🔴 P1 | 消除双方案维护成本 |
| 完全兼容 Next.js 16 | 🔴 P1 | 解决兼容性问题，确保登录可用 |
| 代码简单易懂，易于调试 | 🟡 P2 | 降低维护成本，新人容易上手 |
| 添加速率限制防护 | 🔴 P1 | 防止暴力破解，提高安全性 |
| 添加密码策略验证 | 🟡 P2 | 提高账户安全性 |
| 支持角色权限验证 | 🔴 P1 | 已实现，需要保持 |
| 支持登录/登出完整流程 | 🔴 P1 | 核心功能必须可用 |

### 2.3 方案选择结论

经过分析，**用户/开发团队已选择方案 A：统一使用简化认证**

理由：
1. ✅ **完全兼容 Next.js 16** - 不存在兼容性问题
2. ✅ **代码完全可控** - 调试方便，易于维护
3. ✅ **学习曲线平缓** - 新人容易理解
4. ✅ **开发速度快** - 1-3 天即可完成统一
5. ✅ **风险可控** - 已经有可工作的实现

| 对比维度 | 简化认证 | NextAuth v5 |
|----------|----------|-------------|
| 兼容性 | ✅ 完美兼容 | 🟡 Beta 不稳定 |
| 开发速度 | ✅ 1-3 天 | ❌ 不确定 |
| 维护成本 | ✅ 低 | 🟡 中 |
| 安全性 | ✅ 可自行加固 | ✅ 内置安全 |
| 第三方登录 | 🟡 需要手动实现 | ✅ 开箱即用 |
| 适合场景 | ✅ 当前需求匹配 | 未来需要扩展时考虑 |

## 五、验收标准定义

### 5.1 功能验收标准

---

#### AC-001: 用户登录成功

**前置条件**: 用户已经注册，邮箱 `test@example.com`，密码 `Password123!`

**测试步骤**:
1. 发送 POST 请求到 `/api/auth/login`，body `{ "email": "test@example.com", "password": "Password123!" }`
2. 检查响应
3. 检查响应头中的 Set-Cookie

**预期结果**:
- 响应状态码: `200 OK`
- 响应 JSON: `{ "success": true, "user": { "id": "...", "email": "test@example.com", "name": "...", "role": "..." } }`
- Set-Cookie 包含 `auth-token=...`
- Cookie 属性: `HttpOnly`, `SameSite=lax`, `max-age=604800` (7天)
- 生产环境: `Secure` 属性存在

---

#### AC-002: 用户登录失败 - 密码错误

**测试步骤**:
1. 发送 POST 请求到 `/api/auth/login`，body `{ "email": "test@example.com", "password": "WrongPassword" }`
2. 检查响应

**预期结果**:
- 响应状态码: `401 Unauthorized`
- 响应 JSON: `{ "error": "账号或密码错误" }`
- 未设置 `auth-token` Cookie

---

#### AC-003: 用户登录失败 - 用户不存在

**测试步骤**:
1. 发送 POST 请求到 `/api/auth/login`，body `{ "email": "not-exists@example.com", "password": "any" }`
2. 检查响应

**预期结果**:
- 响应状态码: `401 Unauthorized`
- 响应 JSON: `{ "error": "账号或密码错误" }`

---

#### AC-004: 用户登录失败 - 账户待审批

**前置条件**: 用户状态为 `PENDING_APPROVAL`

**测试步骤**:
1. 发送 POST 请求到 `/api/auth/login`
2. 检查响应

**预期结果**:
- 响应状态码: `401 Unauthorized`
- 响应 JSON: `{ "error": "账户等待审批" }`

---

#### AC-005: 用户登录失败 - 账户已暂停

**前置条件**: 用户状态为 `SUSPENDED`

**测试步骤**:
1. 发送 POST 请求到 `/api/auth/login`
2. 检查响应

**预期结果**:
- 响应状态码: `401 Unauthorized`
- 响应 JSON: `{ "error": "账户已暂停" }`

---

#### AC-006: 获取当前用户 - 已登录

**前置条件**: 用户已登录，`auth-token` Cookie 已设置

**测试步骤**:
1. 发送 GET 请求到 `/api/auth/me`，携带 `auth-token` Cookie
2. 检查响应

**预期结果**:
- 响应状态码: `200 OK`
- 响应 JSON: `{ "authenticated": true, "user": { "id": "...", "email": "...", "name": "...", "role": "..." } }`
- 用户信息与登录用户一致

---

#### AC-007: 获取当前用户 - 未登录

**测试步骤**:
1. 发送 GET 请求到 `/api/auth/me`，不携带 Cookie
2. 检查响应

**预期结果**:
- 响应状态码: `401 Unauthorized`
- 响应 JSON: `{ "authenticated": false }`

---

#### AC-008: 用户登出

**前置条件**: 用户已登录

**测试步骤**:
1. 发送 POST 请求到 `/api/auth/logout`，携带 `auth-token` Cookie
2. 检查响应和 Cookie

**预期结果**:
- 响应状态码: `200 OK`
- 响应 JSON: `{ "success": true, "message": "登出成功" }`
- Set-Cookie 将 `auth-token` 设置为空，`max-age=0`（立即过期）
- 后续调用 `/api/auth/me` 返回 `401 Unauthorized`

---

#### AC-009: 速率限制 - 超过 5 次失败

**测试步骤**:
1. 从同一个 IP 连续发送 6 次错误密码的登录请求
2. 检查第 6 次响应

**预期结果**:
- 第 6 次响应状态码: `429 Too Many Requests`
- 响应包含 `error: "请求过于频繁，请稍后再试"`
- 响应包含 `retryAfter` 字段指示重试时间（秒）
- 响应头包含 `Retry-After`

---

#### AC-010: 速率限制 - 锁定后等待重置

**测试步骤**:
1. 触发速率限制锁定
2. 等待重置时间过后
3. 再次尝试登录

**预期结果**:
- 等待后可以再次尝试登录
- 正确的密码可以登录成功

---

#### AC-011: 权限验证 - 已认证通过

**前置条件**: API 路由使用 `requireAuth`

**测试步骤**:
1. 已登录用户发送请求到受保护的 API
2. 检查响应

**预期结果**:
- 请求通过认证，正常执行业务逻辑
- 不返回 401 错误

---

#### AC-012: 权限验证 - 未认证拒绝

**前置条件**: API 路由使用 `requireAuth`

**测试步骤**:
1. 未登录用户发送请求到受保护的 API
2. 检查响应

**预期结果**:
- 响应状态码: `401 Unauthorized`
- 返回错误信息: `{ "success": false, "error": "未认证", "message": "请先登录" }`

---

#### AC-013: 角色验证 - 权限足够通过

**前置条件**: API 需要 `MANAGER` 角色，当前用户是 `ADMIN`

**测试步骤**:
1. 用户发送请求
2. 检查结果

**预期结果**:
- 请求通过，正常处理
- 不返回 403 错误

**角色等级规则**: `ADMIN` > `MANAGER` > `USER` > `VIEWER`，高等级可以访问低等级资源

---

#### AC-014: 角色验证 - 权限不足拒绝

**前置条件**: API 需要 `MANAGER` 角色，当前用户是 `USER`

**测试步骤**:
1. 用户发送请求
2. 检查结果

**预期结果**:
- 响应状态码: `403 Forbidden`
- 返回错误信息: `{ "success": false, "error": "权限不足", "message": "需要 MANAGER 或更高权限" }`

---

#### AC-015: 刷新页面保持登录

**前置条件**: 用户已登录

**测试步骤**:
1. 在浏览器登录成功
2. 刷新页面
3. 检查是否仍然保持登录

**预期结果**:
- 刷新页面后，调用 `/api/auth/me` 仍然返回用户信息
- 用户不需要重新登录

---

#### AC-016: 密码加密存储

**前置条件**: 用户注册时设置密码

**验证步骤**:
1. 查看数据库中 `passwordHash` 字段
2. 检查是否为 bcrypt 哈希格式

**预期结果**:
- 数据库不存储明文密码
- `passwordHash` 是 bcrypt 哈希 (`$2b$...` 格式)
