## 1. 用户需求深度分析

### 1.1 各角色日常工作流程

#### 1.1.1 创始人 / 老板

**日常工作：**
- 查看整体业务数据报表（营收、利润、库存）
- 审批大额采购、付款
- 审批特殊折扣价格
- 审核新品开发计划
- 管理团队权限
- 对接供应商谈判
- 财务对账

**每周/每月：**
- 月度业绩复盘
- 制定下月销售目标
- 供应商付款计划审批

---

#### 1.1.2 运营经理

**日常工作：**
- 监控各平台店铺数据
- 调整产品价格和推广策略
- 协调业务员、美工、采购工作
- 处理平台客户纠纷
- 库存补货计划制定
- 广告投放预算控制

**每周/每月：**
- 销售数据汇总报表
- 库存盘点核对
- 新品上架计划安排

---

#### 1.1.3 业务员（3 人）

**日常工作：**
- 阿里国际站询盘回复
- 客户开发与跟进
- 报价单制作与发送
- 订单合同签订
- 客户回款跟踪
- 样品申请

**每周/每月：**
- 业绩目标达成跟踪
- 客户跟进记录更新
- 下周工作计划

---

#### 1.1.4 美工（2 人）

**日常工作：**
- 产品主图设计
- 详情页制作
- 平台活动图片设计
- 视频剪辑（TikTok/独立站）
- 图片版权管理

**每周/每月：**
- 新品图片交付计划
- 现有图片优化更新

---

#### 1.1.5 采购（1 人）

**日常工作：**
- 根据订单需求向供应商下单
- 跟踪供应商生产进度
- 对接物流货代
- 入库质检核对
- 供应商对账
- 采购价格记录更新

**每周/每月：**
- 原材料备货计划
- 供应商对账单核对
- 付款申请提交

### 1.2 痛点与效率瓶颈识别

| 角色 | 痛点描述 | 影响程度 | 建议解决方案 |
|------|----------|----------|--------------|
| **创始人** | 数据分散在各个平台，需要手动汇总报表，耗时且容易出错 | 🔴 高 | 统一数据看板，自动生成多维度报表 |
| | 审批流程不规范，微信/邮件审批容易遗漏 | 🔴 高 | 系统内置审批流，实时提醒 |
| **运营经理** | 库存不准确，超卖断货时有发生 | 🔴 高 | 多平台库存同步，自动预警 |
| | 多平台数据需要分别登录查看，操作繁琐 | 🟡 中 | 统一后台多平台订单同步 |
| **业务员** | 客户信息分散在Excel，跟进不及时丢单 | 🔴 高 | CRM 客户管理，跟进提醒 |
| | 报价单需要手动计算价格，容易出错 | 🟡 中 | 系统自动计算，模板导出 |
| **采购** | 采购价格历史无记录，难以议价对比 | 🔴 高 | 供应商价格库，历史趋势分析 |
| | 海外仓备货计划靠经验，容易积压或缺货 | 🔴 高 | 基于销售数据智能备货建议 |
| **美工** | 设计文件管理混乱，找图耗时间 | 🟡 中 | 素材资源库管理，按产品分类 |
| **全体** | 多平台需要分别登录，账号密码管理麻烦 | 🟢 低 | SSO 单点登录 |

**核心痛点总结：**
1. **数据孤岛**：各平台数据分散，无法统一查看
2. **流程不规范**：审批无记录，依赖人工跟进容易遗漏
3. **库存不准确**：多平台库存不同步，超卖/断货
4. **客户流失**：客户信息未系统化管理，跟进不及时
5. **采购盲猜**：备货靠经验，缺乏数据支撑

### 1.3 多平台业务差异分析

| 维度 | 阿里国际站（现有） | 亚马逊（规划） | TikTok（规划） | 独立站（规划） |
|------|-------------------|---------------|---------------|----------------|
| **订单类型** | 大额批发 (MOQ) | FBA/自发货 小额多单 | 小额批发/DTC | DTC 零售 |
| **付款方式** | T/T 信用证 | 平台收款 | PayPal/信用卡 | PayPal/信用卡 |
| **价格体系** | 阶梯报价 | 固定定价 | 活动定价 | 动态定价 |
| **库存管理** | 国内仓直发为主 | FBA 海外仓 | 海外仓一件代发 | 海外仓一件代发 |
| **客户管理** | B 端企业客户 | 终端消费者 | 混合 | 终端消费者 |
| **售后** | 较少纠纷 | 退换货频繁 | 退换货多 | 退换货多 |
| **对账周期** | 月结/款到发货 | 周期结算 | 周期结算 | 周期结算 |
| **核心需求** | 询盘转化、CRM | FBA 库存管理 | 内容标签同步 | 订单同步 |

**关键差异总结：**
- **B 端 vs C 端**：阿里国际站是 B 端批发，其他三个偏向 C 端/小 B
- **仓储模式**：阿里以国内直发为主，亚马逊/TikTok/独立站需要海外仓一件代发
- **订单规模**：阿里单量大、单量少；其他平台单量多、单量小
- **价格策略**：B 端需要阶梯报价、折扣审批；C 端需要灵活定价

## 2. 权限体系设计

### 2.1 角色定义

基于 10 人团队规模，设计以下角色：

| 角色名称 | 人数 | 说明 |
|----------|------|------|
| 超级管理员 | 1 | 创始人，拥有所有权限 |
| 运营经理 | 1 | 负责整体运营，数据分析，库存计划 |
| 业务员 | 3 | 客户开发，询盘跟进，报价，订单 |
| 美工 | 2 | 设计制作，素材管理 |
| 采购 | 1 | 采购下单，供应商管理，备货 |
| 财务 | 0 (预留) | 未来扩展 |

### 2.2 用户注册审批流程

```
用户提交注册申请
        ↓
超级管理员收到审批提醒
        ↓
超级管理员审核：
  ✅ 通过 → 分配角色 → 账号激活
  ❌ 拒绝 → 填写拒绝理由 → 通知申请人
        ↓
完成
```

**规则：**
- 新用户必须经过超级管理员审批才能激活
- 支持批量审批
- 保留审批记录日志
- 支持邀请注册链接（超级管理员生成）

### 2.4 数据隔离策略

#### 2.4.1 客户数据隔离

**策略：** 基于负责人隔离
- 业务员只能查看、编辑自己负责的客户
- 运营经理和超级管理员可以查看所有客户
- 支持客户分配转移，转移后原负责人无法查看
- 支持团队共享客户（可选配置）

#### 2.4.2 订单数据隔离

**策略：** 基于客户归属隔离
- 业务员只能查看自己客户产生的订单
- 运营/采购/管理员可以查看所有订单
- 报价单遵循同样规则

#### 2.4.3 价格成本数据隔离

**策略：** 敏感数据权限控制
- 采购成本对业务员透明？→ **配置化**：默认业务员只能看到售价，看不到采购成本
- 创始人/运营/采购可以看到采购成本
- 毛利率数据仅对创始人/运营可见

#### 2.4.4 财务数据隔离

- 付款金额、对账单仅对创始人/采购可见
- 业务员只能看到自己订单的应收款，看不到公司整体财务数据

## 3. 外贸审批流程设计

### 3.1 采购审批（金额分级）

**分级标准：**

| 采购金额 | 审批流程 | 审批人 |
|----------|----------|--------|
| < 5000 元 | 采购申请 → 运营经理审批 → 完成 | 运营经理 |
| 5000 - 50000 元 | 采购申请 → 运营经理初审 → 创始人审批 → 完成 | 运营经理 → 创始人 |
| > 50000 元 | 采购申请 → 运营经理初审 → 创始人终审 → 完成 | 创始人 |

**流程图：**

```
采购创建采购申请
        ↓
系统判断金额区间
        ↓
< 5000 → 运营审批 → 通过 → 执行采购
                  ↓
                 拒绝 → 退回给采购
        ↓
5000 - 50000 → 运营审批 → 通过 → 创始人审批 → 通过 → 执行采购
                                  ↓
                             拒绝 → 退回
        ↓
> 50000 → 运营预审 → 创始人审批 → 通过 → 执行采购
```

**业务规则：**
- 采购不能审批自己发起的申请
- 支持审批前修改撤回
- 支持添加审批意见
- 超期未审批自动提醒（微信/钉钉/邮件）
- 所有审批记录永久留存

---

### 3.2 价格审批（折扣权限）

**折扣分级：**

| 折扣幅度 | 相对于标准价 | 审批人 |
|----------|-------------|--------|
| 折扣 ≥ 95 折 | 优惠 ≤ 5% | 业务员自动通过 |
| 85 折 - 94 折 | 优惠 6%-15% | 运营经理审批 |
| < 85 折 | 优惠 > 15% | 创始人审批 |

**特殊情况：**
- 老客户返单折扣可由运营审批
- 样品单折扣规则同上
- 大MOQ批量订单可以申请更大折扣，必须创始人审批

**流程图：**

```
业务员创建报价
        ↓
计算折扣幅度
        ↓
≥ 95 折 → 自动通过 → 可发送给客户
        ↓
85-94 折 → 运营经理审批 → 通过 → 发送客户
                        ↓
                       拒绝 → 退回修改
        ↓
< 85 折 → 运营初审 → 创始人审批 → 通过 → 发送客户
```

---

### 3.3 订单审批（大额订单）

**定义：** 大额订单 = 单订单金额 > 100,000 元

**审批流程：**

```
业务员创建订单
        ↓
判断订单金额
        ↓
≤ 100,000 → 自动通过 → 安排采购/生产
        ↓
> 100,000 → 创始人审批 → 通过 → 安排生产
                        ↓
                       拒绝 → 退回业务员
```

**业务规则：**
- 收到预付款的大订单可直接通过，无需审批
- 信用证付款的大订单必须审批
- 新客户首单大额必须审批

---

### 3.4 付款审批（分级授权）

**金额分级：**

| 付款金额 | 审批流程 | 审批人 |
|----------|----------|--------|
| < 10000 元 | 付款申请 → 运营经理审批 → 付款 | 运营经理 |
| 10000 - 100000 元 | 付款申请 → 运营初审 → 创始人审批 → 付款 | 运营 → 创始人 |
| > 100000 元 | 付款申请 → 运营初审 → 创始人终审 → 付款 | 创始人 |

**流程图：**

```
采购创建付款申请
  ↓
  附上对账单/发票
  ↓
金额判断
  ↓
< 10000 → 运营审批 → 通过 → 财务付款
                    ↓
                   拒绝 → 退回
  ↓
≥ 10000 → 运营初审 → 通过 → 创始人审批 → 通过 → 付款
```

**业务规则：**
- 付款必须附件对应对账单/发票（支持上传PDF/JPG）
- 同一供应商月度合并付款，按总金额走审批
- 紧急付款可标记加急，优先提醒审批人
- 支持批量付款申请，按单笔金额分别走审批

**特殊规则：**
- 运费、货代费用由采购申请，运营审批（< 10000）
- 样品费小额采购走快速审批通道

## 6. 验收标准

### 6.1 功能验收标准

| 验收项 | 验收标准 | 测试方法 |
|--------|----------|----------|
| **权限隔离** | 业务员登录后只能看到自己的客户，看不到同事的 | 使用业务员账号登录，尝试访问同事客户数据 |
| **分级审批** | 不同金额走不同审批流，正确到达对应审批人 | 创建不同金额申请，检查审批节点是否正确 |
| **折扣审批** | 折扣超过权限自动进入审批，不允许直接发送 | 创建不同折扣报价，检查是否正确触发审批 |
| **库存更新** | 采购入库后库存数量正确更新 | 创建采购单，入库前后检查库存变化 |
| **数据报表** | 报表数据和明细数据对得上，统计正确 | 对比报表汇总和订单明细汇总 |
| **审批提醒** | 有新审批时，审批人能收到通知 | 创建审批，检查审批人是否收到提醒 |
| **文件上传** | 支持PDF/图片附件上传，可下载查看 | 上传测试文件，尝试下载 |

### 6.2 性能验收标准

- 页面响应时间 < 2 秒
- 支持 10 人同时在线使用无明显卡顿
- 数据导出在 1 分钟内完成
- 支持 10 万+ 客户数据存储

### 6.3 安全验收标准

- 密码加密存储
- 操作日志留存 1 年以上
- 敏感数据（成本、财务）按权限隐藏
- 越权访问测试不能通过
