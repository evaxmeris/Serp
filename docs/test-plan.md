## 🎯 测试目标

本测试计划旨在确保 Trade ERP 系统的**订单管理**和**采购管理**模块满足以下目标：

### 功能目标
- ✅ 验证订单创建、编辑、删除、查询功能正常工作
- ✅ 验证采购单创建、编辑、删除、查询功能正常工作
- ✅ 验证订单状态流转符合业务逻辑
- ✅ 验证采购单状态流转符合业务逻辑
- ✅ 验证订单与客户的关联关系正确
- ✅ 验证采购单与供应商的关联关系正确
- ✅ 验证订单商品明细计算准确
- ✅ 验证采购商品明细计算准确

### 质量目标
- ✅ 关键功能测试覆盖率 ≥ 95%
- ✅ API 接口测试覆盖率 ≥ 90%
- ✅ 严重 bug 数 = 0
- ✅ 高优先级 bug 修复率 = 100%
- ✅ API 响应时间 < 500ms (95% 请求)

## 🧪 测试策略

### 测试类型

#### 1. 单元测试 (Unit Testing)
- **目标**: 验证单个函数/方法的正确性
- **工具**: Jest / Vitest
- **覆盖范围**: 
  - 金额计算函数
  - 状态验证函数
  - 数据验证函数
- **负责人**: 开发团队

#### 2. 集成测试 (Integration Testing)
- **目标**: 验证模块间接口和数据流
- **工具**: Jest + Supertest
- **覆盖范围**:
  - API 端点测试
  - 数据库操作测试
  - 模块间依赖测试
- **负责人**: QA 团队

#### 3. 端到端测试 (E2E Testing)
- **目标**: 验证完整用户流程
- **工具**: Playwright
- **覆盖范围**:
  - 用户创建订单完整流程
  - 用户创建采购单完整流程
  - 状态流转完整流程
- **负责人**: QA 团队

#### 4. API 测试
- **目标**: 验证 REST API 功能
- **工具**: curl / Postman / Jest
- **覆盖范围**: 所有订单和采购相关 API
- **负责人**: QA 团队

#### 5. 性能测试
- **目标**: 验证系统在高负载下的表现
- **工具**: k6 / Artillery
- **覆盖范围**:
  - 并发订单创建
  - 大数据量查询
- **负责人**: 运维团队

### 测试数据策略

#### 测试数据准备
```sql
-- 测试客户数据
INSERT INTO customers VALUES (...);

-- 测试供应商数据
INSERT INTO suppliers VALUES (...);

-- 测试产品数据
INSERT INTO products VALUES (...);
```

#### 数据清理
- 每次测试前清空测试数据
- 使用事务回滚确保数据隔离
- 测试数据前缀：`TEST_`

## ✅ 质量标准

### 准入标准 (Entry Criteria)

- [ ] 开发完成并通过单元测试
- [ ] 代码审查通过
- [ ] 测试环境就绪
- [ ] 测试用例评审通过

### 准出标准 (Exit Criteria)

- [ ] 所有 P0/P1 测试用例执行完毕
- [ ] 严重 bug 数 = 0
- [ ] 高优先级 bug 修复率 = 100%
- [ ] 测试覆盖率 ≥ 90%
- [ ] 性能指标达标
- [ ] 产品经理验收通过

### 缺陷优先级定义

| 优先级 | 定义 | 修复时限 |
|--------|------|----------|
| **P0 - 严重** | 系统崩溃、数据丢失、核心功能不可用 | 24 小时 |
| **P1 - 高** | 主要功能异常、影响业务流程 | 48 小时 |
| **P2 - 中** | 次要功能异常、有替代方案 | 1 周 |
| **P3 - 低** | UI 问题、体验优化 | 下个版本 |
